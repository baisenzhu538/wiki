# Kimi 深度调研集群方法论 (Deep-Research-Swarm)

# Kimi 深度调研集群方法论 (Deep-Research-Swarm)

> **核心理念**: 多智能体认知三角测量（Epistemic Triangulation）——在不同研究维度上发散、检测重叠与矛盾、深度验证、再收敛为经过验证的综合结论。集群并行服务于认知鲁棒性，而非仅仅追求速度。

---

## 一、自适应路由体系 (Phase 0)

所有调研任务首先经过**意图与输入路由器**，根据任务特征自动分类为四条路线：

| 路线 | 适用场景 | 特征 |
|------|---------|------|
| **Route A — 广域搜索** | 宽泛/探索性主题，维度不明确 | 先广域探索，再深度挖掘；两阶段并行集群 |
| **Route B — 聚焦搜索** | 具体问题，维度清晰 | 标准流程：概览扫描 → 分解 → 并行深度挖掘 |
| **Route C — 纯文件研究** | 用户明确要求仅基于上传文件 | **零外部搜索**；多文件深度分析 → 交叉验证 → 写作 |
| **Route D — 文件增强研究** | 文件为主要参考，允许外部补充 | 先分析文件，再针对空白进行外部搜索，融合输出 |

**分类信号**：
- 文件 + "仅基于文件" / "不上网搜索" → **Route C**
- 文件 + "参考这些文件" / "结合文件" / 无明确限制 → **Route D**
- 无文件 + 宽泛/探索性主题 → **Route A**
- 无文件 + 具体/有界问题 → **Route B**

---

## 二、完整工作流架构

```
用户查询
  │
  ▼
Phase 0: 意图与输入路由
  │
  ├─ Route A: 广域搜索
  │   → Phase 1 (快速概览) → Phase 1W (多Agent广域探索)
  │     → Phase 2 (维度分解) → Phase 3 (并行深度挖掘)
  │       → Phase 4 (交叉验证) → Phase 5 (如有冲突)
  │         → Phase 6 (洞察提取) → Phase 7 (移交写作)
  │
  ├─ Route B: 聚焦搜索
  │   → Phase 1 (概览扫描) → Phase 2 (维度分解)
  │     → Phase 3 (并行深度挖掘) → Phase 4 (交叉验证)
  │       → Phase 5 (如有冲突) → Phase 6 (洞察提取) → Phase 7
  │
  ├─ Route C: 纯文件研究
  │   → Phase F (文件 intake 与深度分析)
  │     → Phase 2 (从文件主题分解维度)
  │       → Phase 3-F (多Agent文件深度挖掘，无外部搜索)
  │         → Phase 4 (跨文件交叉验证) → Phase 6 → Phase 7
  │
  └─ Route D: 文件增强研究
      → Phase F (文件 intake 与深度分析)
        → Phase 1 (针对性概览，填补文件空白)
          → Phase 2 (融合文件主题 + 外部概览分解维度)
            → Phase 3 (并行深度挖掘，文件上下文 + 搜索)
              → Phase 4 → Phase 5 → Phase 6 → Phase 7
```

## 三、各阶段详细方法论

### 3.1 认知重置规则 (Epistemic Reset Rule)

在任何分析或叙述生成之前，系统必须：
- **假设内部知识可能过时或不完整** —— 始终先获取当前日期/时间
- **时间敏感性约束** —— 当用户查询含时间要求（如"2026 Q1"、"最近6个月"），将指定时间范围视为硬约束，搜索查询必须 targeting 该窗口
- **外部证据优先** —— 先进行外部广泛搜索建立证据图景（Route C 除外）
- **禁止先搜索后叙事** —— 在搜索/文件分析输出前不得生成任何事实性声明
- **搜索语言规则** —— 所有搜索查询必须与用户消息语言一致（中文查询用中文搜索，英文查询用英文搜索）
- **所有输出必须使用内联引用 `[^number^]` 标注原始来源**

---

### 3.2 Phase F: 文件 Intake 与深度分析 (Route C & D)

**目标**: 从所有上传文件中提取结构化知识，构建证据地图，识别主题、声明、矛盾与空白。

**流程**: 
1. **文件清点**: 列出所有上传文件的类型、大小、一行内容摘要
2. **逐文件提取**（每文件提取）:
   - 核心主题与议题
   - 关键声明、论点、结论
   - 数据点、统计数字、图表（附页码/章节引用）
   - 方法论（如适用）
   - 局限性、注意事项、作者指出的偏见
3. **跨文件映射**:
   - 识别跨文件的重叠主题
   - 检测文件间矛盾或冲突数据
   - 映射互补信息（文件A提供文件B缺失的上下文）
   - 识别**空白** —— 无文件覆盖的重要方面
4. **主题整合**: 产出整合主题列表，供 Phase 2 维度分解使用

**路线差异**:
- **Route C**: 空白分析仅作信息记录（不外部填补空白），整合主题为 Phase 2 唯一基础
- **Route D**: 空白分析直接指导 Phase 1 搜索策略 —— Phase 1 针对这些空白进行外部搜索

**输出**: `/mnt/agents/output/research/{topic}_file_analysis.md`

---

### 3.3 Phase 1: 概览扫描 (Route A, B, D)

**目标**: 在承诺维度分解之前，通过"粗到细"的探索建立基于证据的全局叙事图景。

**路线差异**:
- **Route A (广域)**: 轻量化扫描 —— 仅3-5次搜索用于宏观框架，重活 delegated 给 Phase 1W
- **Route B (聚焦)**: 完整5步粗到细扫描
- **Route D (文件增强)**: 针对性扫描 —— 使用 Phase F 的空白分析指导搜索查询，聚焦文件未覆盖领域，3-5次搜索

**完整流程** (Route B 版本；A/D 按上述调整):

| 层级 | 搜索编号 | 目标 | 查询类型 |
|------|---------|------|---------|
| Level 1 — 宏观概览 | Search 1-2 | 建立全局框架 | 广泛概览、行业报告、高层统计、权威来源总结 |
| Level 2 — 结构映射 | Search 2-4 | 理解组成部分 | 市场结构、主要参与者、监管机构 |
| Level 3 — 新兴议题与张力 | Search 5 | 发现动态信号 | 近期发展、冲突叙事、趋势信号 |

**每次搜索后输出**:
- 关键发现（简洁）
- 识别的主导叙事
- 检测到的争议或冲突声明
- 发现的关键参与者和权威来源
- 需要深入调查的差距

---

### 3.4 Phase 1W: 多Agent广域探索 (Route A 专属)

**目标**: 在承诺维度之前，通过并行子Agent最大化搜索广度。这是广域搜索场景的关键差异化环节。

**流程**:
1. **识别 5-8 个广域探索维度 (Facet)**
   - 相互互补（合起来覆盖完整问题空间）
   - 部分重叠（≥20%重叠用于交叉验证）
   - 维度类型示例: 技术图景、市场/商业图景、监管/政策图景、竞争动态、用户/消费者视角、供应链、地理差异、历史演进、新兴颠覆

2. **并行部署 ≥5 个子Agent**，每个维度一个
   - 每个子Agent的 prompt 必须包含：维度范围、Phase 1 上下文、搜索要求（≥10次独立搜索）、输出格式、输出文件路径

3. **每个广域探索子Agent必须**:
   - 执行 **≥10次独立搜索**，查询多样化（不回收关键词）
   - 广撒网：不同来源类型、维度内不同角度
   - 识别关键参与者、数据点、趋势和争议
   - 标记值得 Phase 3 深入调查的领域
   - 保存输出到 `/mnt/agents/output/research/{topic}_wide{NN}.md`

4. **编排器综合**: 所有广域探索Agent完成后:
   - 读取所有 `{topic}_wide{NN}.md`
   - 合并发现为统一图景地图
   - 识别最有前景和最具争议的领域
   - 将丰富图景输入 Phase 2 进行维度分解

**关键原则**: Phase 1W 追求**广度** —— 发现存在什么、谁重要、正在发生什么。Phase 3 追求**深度** —— 彻底调查每个维度。两阶段集群确保不遗漏重要信息。

---

### 3.5 Phase 2: 维度分解

**目标**: 确定研究维度，准备子Agent任务分配。

**各路线输入**:
- **Route A**: Phase 1 + Phase 1W 广域探索输出（最丰富的输入）
- **Route B**: Phase 1 概览扫描
- **Route C**: Phase F 文件分析 —— 仅整合主题列表（无外部输入）
- **Route D**: Phase F 文件分析 + Phase 1 针对性概览扫描

**分解规则**:
- **≥10个维度（强制性最低要求）**。越多越好 —— 10-20个维度，取决于主题复杂度
- 每个维度从**截然不同的角度或场景**切入，确保研究从根本不同的视角覆盖问题空间
- 维度可按以下方式组织：
  - **分析角度**: 技术、经济、监管、伦理、竞争、用户侧、供应链等
  - **场景**: 乐观、悲观、现状、颠覆、黑天鹅等
  - **利益相关者视角**: 消费者、企业、监管者、投资者、竞争者、劳动力等
  - **地理或细分市场**: 中国、美国、欧盟、新兴市场等
  - **时间跨度**: 历史起源、现状、1年展望、5年展望等
  - **文件衍生主题** (Route C/D): 维度可映射到 Phase F 识别的主要主题
- **≥30% 概念重叠** —— 相关维度间的重叠创造交叉验证压力
- 每个维度必须覆盖：
  1. **当前状态** —— 从此角度正在发生什么，始终带 `[^number^]` 引用
  2. **关键证据** —— 数据、来源、具体实例，使用 `[^number^]`
  3. **张力与反论** —— 从此角度存在什么对立观点，所有声明通过 `[^number^]` 引用

**Route C 特殊规则**: 文件-only 研究中，维度完全从文件主题派生。每个维度应映射到一个或多个文件，范围应引用具体文件章节。

---

### 3.6 Phase 3: 并行深度挖掘 (子Agent部署)

**目标**: 在所有维度上并行执行深度优先研究。**≥10个子Agent同时启动**，每个维度一个。

#### 标准模式 (Route A, B, D)

1. 通过 `task` 创建每个维度一个子Agent —— **所有子Agent并行启动**（不串行化）
2. 每个子Agent从其分配的角度/场景进行调查，产出与其他Agent不同但部分重叠的发现
3. 每个子Agent的 prompt 必须包含：
   - **(1) 任务**: 维度范围、四个要求角度（当前状态、历史、利益相关者、反叙事）、深度期望
   - **(2) 上下文**: 与此维度相关的早期阶段关键发现
   - **(3) 文件上下文 (仅 Route D)**: Phase F 文件分析的相关摘录 —— 子Agent应将文件内容视为主要证据，搜索补充/确证的外部来源
   - **(4) 输出格式**: 证据模板（见下）
   - **(5) 输出文件路径**: 必须保存到 `/mnt/agents/output/research/{topic}_dim{NN}.md`

**子Agent要求 (标准模式)**:
- 执行 **≥20次独立搜索**（无重复关键词循环）
- 尽可能调查原始来源（政府网站、学术期刊、官方备案、主流媒体）
- 追溯声明回到原始出版物
- 识别并记录反论点
- 避免内容农场、匿名博客、SEO聚合器
- **Route D**: 显式引用并建立在文件衍生证据之上。外部搜索填补空白、验证文件声明、增加深度。清楚区分文件来源 vs 搜索来源证据
- **保存输出到 `/mnt/agents/output/research/{topic}_dim{NN}.md`**

#### 纯文件模式 (Route C)

1. 通过 `task` 创建每个维度一个子Agent —— **所有子Agent并行启动**
2. 每个子Agent**仅使用上传文件内容**分析其分配维度 —— **零外部搜索**
3. 每个子Agent的 prompt 必须包含：任务、完整文件内容或相关摘录、分析要求、输出格式、输出文件路径

**子Agent要求 (纯文件模式)**:
- 彻底分析所有与维度相关的提供的文件内容
- 跨多个文件交叉引用声明和数据
- 评估证据质量并识别潜在偏见
- 标记文件证据薄弱或矛盾之处
- 识别来源材料中的隐含假设
- **不得执行任何外部搜索**
- **保存输出到 `/mnt/agents/output/research/{topic}_dim{NN}.md`**

#### 统一子Agent输出格式 (所有模式)

```
Claim: [带内联引用的已识别声明]
Source: [来源名称 / 文件名称]
URL: [来源 URL / "File: {filename}, Section: {section}"]
Date: [出版日期 / 文件用 "N/A"]
Excerpt: [逐字原文摘录 —— 不转述]
Context: [周围上下文]
Confidence: [high / medium / low]
```

### 3.7 Phase 4: 交叉验证引擎 (编排器)

**目标**: 比较所有维度输出，分类置信度， surfaced 矛盾，并**保存验证结果到文件**供下游报告写作使用。

**流程**:
1. 读取所有 `/mnt/agents/output/research/{topic}_dim{NN}.md` 文件
2. 将每个发现分类为四个层级之一：

| 层级 | 标准 |
|------|------|
| **高置信度** | ≥2个Agent从独立来源以一致证据确认 |
| **中等置信度** | 1个Agent从权威来源确认 |
| **低置信度** | 来源薄弱、博客级证据，或单一未验证声明 |
| **冲突区域** | Agent间统计分歧、解释分歧、时间不一致，或同一指标数值差异（如两个Agent报告同一统计数字不同） |

3. **显式列出所有冲突区域项目** —— 矛盾被高亮和分析，从不压制
4. 确定是否需要 Phase 5（任何冲突区域或关键低置信度项目）
   - **Route C 例外**: Phase 5 跳过（不允许外部搜索）。冲突按原样记录并带入 Phase 6
5. 内联引用 `[^number^]` 必须保留
6. 冲突区域分析必须包含 `[^number^]` 引用所有涉及的来源

**输出**: 保存完整交叉验证结果（所有层级 + 冲突区域分析）到 `/mnt/agents/output/research/{topic}_cross_verification.md`

---

### 3.8 Phase 5: 针对性验证 (条件性)

**目标**: 解决冲突，强化薄弱发现。

**触发条件**: 仅当 Phase 4 识别出冲突区域或关键低置信度项目时执行。**Route C 不可用**（纯文件研究无法调用外部搜索）。

**流程**:
1. 对每个未解决项目，部署一个聚焦子Agent，携带：
   - 具体冲突声明及其来源
   - 寻找独立证据解决分歧的指示
   - 每个冲突至少3次额外搜索

2. 重复直到每个项目被：
   - **已解决** —— 以新证据重新分类为高/中置信度
   - **显式标记为未解决** —— 记录为领域内的真实分歧

3. **更新** `/mnt/agents/output/research/{topic}_cross_verification.md`，加入解决结果。

---

### 3.9 Phase 6: 洞察提取

**目标**: 识别非显而易见的洞察 —— 不直接出现在前期发现中，但从跨维度分析中涌现。

**洞察定义**: 洞察是从多个已验证发现中推导出的高层推论。不得重复先前陈述的声明或证据。

**流程**:
1. 审阅 Phase 3-5（及 Phase F 文件分析，如适用）的所有已验证发现
2. 识别仅在比较多个维度时才可见的模式
3. 提取揭示以下方面的洞察：结构关系、隐藏张力、新兴趋势、系统性风险、战略机会
4. 确保每个洞察至少间接受两个维度证据支持

**路线特定侧重**:
- **Route C/D (文件型)**: 优先从跨文件综合中涌现的洞察 —— 无单文件能独自揭示的模式。Route D 还需高亮外部证据在何处强化、矛盾或延伸文件衍生结论
- **Route A (广域)**: 优先桥接不同探索维度的洞察 —— 独立探索领域间的联系

**体裁感知洞察提取**: 根据预期下游写作格式调整侧重：
- **报告**（行业报告、市场分析、咨询交付物）: 优先可执行战略洞察、市场机会、竞争动态、前瞻性启示
- **学术论文**（综述、实证研究、文献回顾）: 优先研究空白、方法论矛盾、理论张力、对先前工作的定位新角度

**每条洞察记录**:
- **Insight**: 推断模式的简洁陈述
- **Derived From**: 维度引用（如 Dim 02, Dim 07）、支持证据集群（Route C/D 包含文件引用）
- **Rationale**: 洞察如何从证据中涌现的解释
- **Implications**: 潜在影响或意义
- **Confidence**: high / medium / exploratory

**输出**: 保存所有洞察到 `/mnt/agents/output/research/{topic}_insight.md`。此文件是整个深度研究过程的核心综合，将成为下游写作技能的主要输入。

**规则**:
- 洞察不得重复现有发现
- 洞察必须来自跨维度比较
- 避免无证据支持的推测性声明
- 最低输出: 5条洞察
- 洞察必须包含内联引用 `[^number^]`

---

### 3.10 Phase 7: 移交写作技能

**目标**: 将所有研究制品移交给适当的写作技能进行最终文档生成。

**流程**:
1. 验证 `/mnt/agents/output/research/` 下所有必需文件存在:
   - `{topic}_dim{NN}.md` —— 所有维度文件（≥10）
   - `{topic}_cross_verification.md` —— 置信度层级和冲突分析
   - `{topic}_insight.md` —— 跨维度洞察
   - `{topic}_file_analysis.md` —— （Route C/D）文件 intake 分析
   - `{topic}_wide{NN}.md` —— （Route A）广域探索输出
2. 根据用户意图确定目标写作技能：
   - `report-writing` —— 行业报告、市场分析、咨询交付物、政策简报
   - `paper-writing` —— 学术论文、综述论文、文献回顾、会议投稿
3. 调用选定写作技能，在移交中提供**显式文件路径**
4. 编排器必须显式告知写作技能深度研究已完成，无需额研究子Agent
5. 最终文档必须纳入 Phase 6 的洞察
6. 如用户指定时间范围，编排器必须在移交前验证最终制品中的关键数据点和日期是否落在该范围内

## 四、搜索预算与 Agent 规模

| 路线 | Agent 规模 | 搜索预算 | 说明 |
|------|-----------|---------|------|
| **Route A** | ≥5广域Agent + ≥10深度Agent | **≥250次搜索** | 广域: 每Agent ≥10次；深度: 每Agent ≥20次 |
| **Route B** | ≥10深度Agent | **≥200次搜索** | 每Agent ≥20次 |
| **Route C** | ≥10文件Agent | **0次外部搜索** | 纯文件分析 |
| **Route D** | ≥10深度Agent | **≥150次搜索** | 每Agent ≥15次（文件提供基础证据，搜索需求减少） |

---

## 五、核心原则

1. **深度优先** (Route B) / **先广度后深度** (Route A)。禁止浅层聚合。每个维度在进入下一阶段前必须被彻底调查。
2. **原始证据必需**。子Agent必须返回带来源URL/文件引用和日期的逐字摘录。禁止仅转述的输出。
3. **矛盾即信号**。冲突被高亮和分析，从不被压制或平均化处理。
4. **一切皆是文件**。长形式研究内容绝不输出在聊天中。聊天仅用于状态更新。
5. **来源质量至上**。优先顺序: 政府网站、学术期刊、官方备案、主流媒体。避免: 内容农场、匿名博客、SEO聚合器。文件型路线中，用户提供的文件视为主要权威来源。
6. **所有输出必须包含 `[^number^]` 风格引用**。
7. **所有文件保存在 `/mnt/agents/output/research/`**。无例外。
8. **Route C 尊重用户意图**。用户说"仅基于文件"时，绝不偷偷加入外部搜索。对用户意图的保真度至上。
9. **Route D 平衡来源**。文件内容为主；外部搜索填补空白并增加深度。不让外部来源掩盖用户提供的材料。
