# Business Research（商业调研 OSCAR 武器库）

将用户提出的商业问题，通过 OSCAR 方法论 + 13 武器体系，转化为可指导决策的情报报告。

## Constraints

<hard_limits>
- 严禁捏造信源：每条事实必须附可验证的 URL 或"来源名称+日期"，无来源者标为 ❓
- 核心结论必须有 ≥2 个独立 L1/L2 来源交叉验证，否则降级标注 ⚠️
- Step 0 用户确认意图前，禁止任何搜索或分析动作
- Step 14 报告生成前必须通过 Step 13 综合质量门（BLOCKING）
- 关键数字（金额/市占率/用户规模）必须点开原始链接核验，不接受模型记忆输出
</hard_limits>

<constraints>
- 时效性：🔴极高敏感领域（AI/监管/融资）信源时效 ≤30天；🟠高敏感 ≤12个月；其余见 Step 1 表格
- 置信度标注：结论只用 ✅高 / ⚠️中 / ❓低，禁止"可能""大概"等无标注模糊词
- 武器覆盖率：标准报告最低 5 个武器，深度尽调最低 10 个
- 搜索优先于模型内置知识；模型内置知识用 🧠 标注，时效截止 2025年5月
- 报告禁用 AI 套话（全方位/赋能/颠覆/引领），换为具体证据陈述（见 references/report-guide.md）
- "不知道"优于"编造"：找不到可靠信息时，标注"未找到可靠来源"，不做推测填充
</constraints>

## 核心理念

- **假设驱动**：调研起点是"验证什么假设"，不是"了解一下"
- **交叉验证**：任何关键结论必须 ≥2 个独立来源确认
- **先博后渊**：先用 20% 时间全局扫描，再用 80% 时间重点突破
- **信源分级**：L1 官方 > L2 权威报告 > L3 媒体 > L4 社区/个人
- **反向思考**：每个结论出炉前先做 Pre-Mortem——"假设这结论是错的，为什么？"
- **穷尽意识**：对每个节点追问"还有吗"直到真正穷尽

## OSCAR 方法论

| 环节 | 含义 | 关键动作 |
|------|------|---------|
| **O** — Objective | 明确调研目标 | 用一句话写清楚要回答什么决策问题 |
| **S** — Scope | 缩小调研范围 | 明确边界：必须获取 / nice-to-have / 不在范围内 |
| **C** — Checklist | 罗列信息清单 | 细化到具体信息项，而非"了解竞品情况" |
| **A** — Acquisition | 穷尽手段获取 | 运用 13 武器体系 |
| **R** — Reasoning | 分析判断归因 | 交叉验证、信源分级、假设验证 |

## Trigger Conditions

- 调研某家公司、品牌或产品
- 分析某个行业、赛道或市场
- 做竞品分析或竞争格局梳理
- 评估加盟/投资/合作项目
- 收集商业情报辅助决策

**与其他 Skill 的区分**：
- 纯技术/概念/学术调研 → `/deep-research`
- 需要演示文稿 → `/remotion-slides`
- 需要可视化图表 → `/text-visualizer`
- 需要完整 PDF 手册 → `/huanwangorg-book-pdf`

## Step 0: Preflight 用户确认（BLOCKING）

**在开始任何调研动作前，必须先与用户对齐。**

```
在开始调研前，我需要确认几个方向：

1. **调研类型**：
   - 🏢 企业调研（单个公司的全面摸底）
   - ⚔️ 竞品分析（2-5 家企业的对比分析）
   - 📊 行业/赛道分析（市场规模、格局、趋势）
   - 🔍 项目评估（加盟/投资/合作的可行性评估）
   - 🎯 专项调研（特定问题，如定价策略、用户画像）

2. **调研深度**：
   - 📋 快速摸底（2-4小时级，核心信息 + 关键判断）
   - 📊 标准调研（1-2天级，系统分析 + 多维对比）
   - 🔬 深度尽调（3-5天级，穷尽渠道 + 交叉验证 + 专家视角）

3. **决策场景**：这次调研要支撑什么决策？

4. **你已知的信息**：关于这个主题，你已经知道什么？有什么初步判断？
```

用户回复后：记录到 `00_调研设计.md`；如用户说"默认"，直接进入 Step 1，不再追问。

**调研深度影响**：

| 深度 | 假设数量 | 武器覆盖 | 是否并行 |
|------|---------|---------|---------|
| 快速摸底 | 1-2 个 | 3-5 个 | 否 |
| 标准调研 | 3-5 个 | 7-9 个 | 建议 |
| 深度尽调 | 5-8 个 | 10+ 个 | 必须 |

## Step 1: 假设构建与边界界定（O + S of OSCAR）

### 1.1 将模糊问题转化为可证伪假设

把用户的模糊问题拆解为 **2-5 个可验证假设**：

| 模糊问题 | 转化后的假设 |
|---------|-------------|
| "这个赛道靠谱吗" | H1: 市场规模>X亿且年增速>Y% / H2: 头部玩家市占率<40%（还有空间）/ H3: 获客成本在可接受范围内 |
| "要不要加盟这个品牌" | H1: 单店模型回本周期<18个月 / H2: 现有加盟商满意度>70% / H3: 品牌方合规无重大法律风险 |
| "XX公司实力如何" | H1: 核心业务持续增长 / H2: 团队和技术有竞争壁垒 / H3: 财务健康、现金流充足 |

按优先级排序：
- **致命假设**（如不成立则整个方案推倒）→ 最先验证
- **核心假设**（影响决策方向）→ 其次
- **辅助假设**（nice-to-have）→ 最后

### 1.2 研究边界界定

| 维度 | 必须明确 | 示例 |
|------|---------|------|
| **行业** | 调研哪个细分赛道？ | 新茶饮 vs 咖啡 vs 整个现制饮品 |
| **地域** | 哪个市场？ | 中国大陆 / 东南亚 / 全球 |
| **时间** | 关注哪个时间段？ | 近 3 年 / 近 1 年 / 实时 |
| **企业规模** | 什么体量的公司？ | 头部上市公司 / 初创企业 / 全覆盖 |
| **决策视角** | 以什么身份做决策？ | 投资人 / 创业者 / 加盟商 / 企业高管 |

### 1.3 时效敏感性判断

| 领域 | 敏感级别 | 时间窗口 |
|------|---------|---------|
| AI/加密/新消费 | 🔴 极高 | 3-6 个月 |
| SaaS/电商/社交 | 🟠 高 | 6-12 个月 |
| 传统消费/制造业 | 🟡 中 | 1-2 年 |
| 基础设施/公共服务 | 🟢 低 | 2-3 年 |

### ⛔ Step 1 确认门（BLOCKING）

向用户发出假设确认卡，等确认后再进入 Step 2：

```
**决策问题**：[一句话]
**核心假设**：
  H1 [致命]：[假设描述]
  H2 [核心]：[假设描述]
  H3 [辅助]：[假设描述]
**研究边界**：[行业] × [地域] × [时间] × [企业规模]
这个方向是否符合你的预期？
```

**📁 保存动作**：写入 `00_调研设计.md`

**线下武器**（武器1/2/3/5/6/8/10）：AI 提供行动模板，详见 `references/weapon-action-templates.md`

### 2.2 多 Agent 并行决策

- 快速摸底：不并行
- 标准调研（2-3 Agent）：Agent A（企业面）/ B（市场面）/ C（用户面）
- 深度尽调（3-5 Agent）：A（基础信息）/ B（竞争格局）/ C（产品口碑）/ D（人才组织）/ E（合规风险）

Agent 结果合并：重复事实保留信源评分更高版本；矛盾事实标注「待裁定」进入信息冲突裁决流程。

**📁 保存动作**：武器决策表（13 行）+ Agent 分工追加到 `00_调研设计.md`

## Step 3: 在线信息采集（A of OSCAR - 第一轮）

### AI 反幻觉协议（BLOCKING — 贯穿所有搜索）

1. **搜索优先**：永远先搜索，不从模型记忆生成事实
2. **强制引用**：每条事实附出处 URL；无来源 → 标为 ❓ 低置信度
3. **核心数据核验**：金额/市占率/用户规模必须点开原始链接

### 搜索工具选择

```
实时新闻/动态          → tavily-search（加 time_range 参数）
提取已知 URL           → tavily-extract 或 WebFetch
企业工商/财务           → WebSearch + 天眼查/企查查
学术/专利              → WebSearch + site:限定
AI 深度初筛            → Perplexity Deep Research / Gemini Deep Research
兜底                  → WebSearch
```

优先数据库索引见 `references/databases-index.md`。

### 关键词育种（搜索前必做）

对每个调研子目标，生成**三组搜索关键词**：

| 关键词组 | 目的 | 示例（调研"蔚来"） |
|--------|------|---------------|
| **宽泛组** | 探索边界，发现意外线索 | "新能源汽车 竞争格局" |
| **精确组** | 聚焦核心，获取关键数据 | "蔚来 2024 毛利率" "蔚来 换电站 数量" |
| **长尾组** | 挖掘细节，找到用户真实声音 | "蔚来换电站 用户体验 吐槽" "蔚来车主 续费率" |

> 每组关键词同时准备中英文版本（如"蔚来换电" + "NIO battery swap"）。

### 执行顺序

```
1. 企业基础信息（武器9+11）→ 工商/股东/融资/公开财务
2. 行业全局认知（武器12）→ 行业报告/市场规模/格局/趋势
3. 产品与口碑（武器4+9）→ 产品线/定价/用户评价/投诉
4. 人才与组织信号（武器7）→ 招聘分析/团队规模/技术栈
5. 趋势与热度（武器11+13）→ 搜索指数/新闻动态/社媒声量
6. 社区声音（武器4+13）→ 知乎/小红书/脉脉讨论
```

### 线索驱动侦察循环

```
选线索 → 匹配武器 → 执行采集（记录 URL+查询词）
→ 立即交叉验证关键信息 → 提取新线索加回线索库
→ 标注冲突信息「待裁定」

退出条件（满足任一）：
  a. 完成 3 轮完整循环
  b. 连续 2 轮无高价值新线索
  c. 用户指示停止
```

每条信息**立即**追加到 `01_信息采集日志.md`：

```markdown
## 采集 #[序号]
- **渠道**：[武器编号 + 具体渠道]
- **信息内容**：[摘要]
- **来源**：[URL]
- **信源层级**：L1/L2/L3/L4/L5
- **时效**：[YYYY-MM-DD] ✅当前有效 / ⚠️需验证 / ❌已过时
- **关联假设**：H[编号]
```

## Step 4: 子方向深度委托（可选，用户确认后执行）

**触发条件**：在线采集时，发现某个子方向属于**纯技术/概念/学术深度**（非商业主体），本 Skill 无法充分覆盖。

典型场景：
- 调研"某 AI 公司"时，需深入理解其技术架构（如 MoE、推测解码等）
- 调研"某生物科技企业"时，需解读核心算法论文
- 调研"某行业"时，需要对"技术原理"做系统综述

### 委托流程

```
1. 识别触发：主 Skill 在 Step 3 发现技术深度子方向
2. 向用户确认：
   > 🔄 检测到子方向 [XX] 需要技术深度调研（超出商业调研范畴）。
   >    建议切换 `/deep-research` 单独处理，完成后回到本 Skill 自动合并。
   >    是否委托？（y/n）
3. 用户同意 → 写入交接文件 handoff/deep-research-request.json
4. 用户执行 /deep-research，完成后输出到 handoff/deep-research-result.json
5. 回到本 Skill：Step 6 会自动读入并合并到 02_事实卡片库.md
```

### 交接 JSON 契约

**请求**（`handoff/deep-research-request.json`）：

```json
{
  "source_skill": "business-research",
  "target_skill": "deep-research",
  "topic": "[技术子方向主题]",
  "parent_hypothesis": "H[编号]",
  "decision_context": "[本次商业调研的大目标]",
  "expected_output": {
    "format": "fact_cards",
    "min_citations": 5,
    "source_levels_required": ["L1", "L2"]
  }
}
```

**结果**（`handoff/deep-research-result.json`）：

```json
{
  "source_skill": "deep-research",
  "target_hypothesis": "H[编号]",
  "key_findings": [
    {"fact": "...", "source": "URL", "source_level": "L1", "confidence": "high"}
  ],
  "overall_confidence": "high|medium|low",
  "blind_spots": ["..."]
}
```

Step 6 事实抽取规则：**若 handoff/ 有 `*-result.json`，先读入并合并到事实卡片库，标注 `来源 Skill: deep-research`**。

## Step 6: 事实抽取与信源分级

### 信源评分（五级 + 动态调整）

| 级别 | 信源类型 | 基础权重 |
|------|---------|--------|
| **一级** | 年报/法院判决/政府统计/源码库 | **5** |
| **二级** | 学术论文/顶级券商研报/专家专访 | **4** |

## Step 8: 反向证据强制搜索门（BLOCKING）

**对抗确认偏误的硬机制**：对每个判定为 ✅ 成立的**致命假设**，强制执行一轮反向搜索。

### 执行规则

```
对每个 ✅ 致命假设 Hi：

1. 构造反向搜索词（至少 3 条）：
   - "[假设核心词] 失败"
   - "[假设核心词] 反例 / 质疑 / 争议"
   - "[假设核心词] 倒闭 / 退出市场 / 踩坑"
   - 英文版本（国际化场景）

2. 执行反向搜索：
   - 优先搜 L1/L2 信源（财新/彭博深度报道、学术论文、行业权威）

   - 找到 ≥1 条权威反例（L1/L2）→ 假设降级为 ⚠️ 部分成立
```

## Step 10: 深度量化自评（BLOCKING — Layer 1-3 评分）

**对抗"浅尝辄止"的硬机制**。调研不是"找到了数据"就算深度，要追问"为什么"和"所以呢"。

### Layer 1-3 评分标准

| 层级 | 评分标准 | 分数 | 示例（调研"蔚来毛利率"）|
|------|---------|------|----------------------|
| **Layer 1：What**（表面事实）| 有数据支撑的事实陈述 | **1** | "蔚来 Q3 毛利率 -22.3%" |
| **Layer 2：Why**（原因分析）| 能解释驱动机制/底层逻辑 | **2** | "毛利率为负因 BOM 成本占比 83%、产能利用率仅 40%、折扣促销抢量" |
| **Layer 3：So What**（决策含义）| 能给出对用户决策的具体影响 | **3** | "按现金消耗速度 680亿/18个月，若 Q1 未能融资，存在流动性风险；对投资人意味着需观察 Q4 融资进展" |

### 评分输出（BLOCKING）

在 `04_竞争格局图.md` 末尾追加：

```markdown
## 深度自评（Layer 1-3）

| 假设 | Layer 1 What | Layer 2 Why | Layer 3 So What | 最高层级 |
|------|:---:|:---:|:---:|:---:|
| H1 [致命]| ✅ | ✅ | ✅ | **3** |
| H2 [致命]| ✅ | ✅ | ❌ | 2 |
| H3 [核心]| ✅ | ❌ | ❌ | 1 |

**平均层级**：(3 + 2 + 1) / 3 = **2.0** ≥ 2.0 ✅ 达标
```

### 硬门槛


## Step 11: 交叉验证 + 置信度聚合 + Pre-Mortem（R of OSCAR - 第二轮）

### 11.1 交叉验证规则

| 信息重要性 | 最低独立来源数 | 来源层级要求 |
|-----------|--------------|------------|
| 核心结论依据 | **≥2 个** | 至少 1 个 L1/L2 |
| 次要支撑 | ≥1 个 | L1-L3 均可 |
| 背景信息 | 1 个 | 无限制 |

**矛盾信息处理**：时间新旧 → 采信新；信源级别高低 → 采信高；同级矛盾 → 标「争议信息」，报告中如实呈现。

### 11.2 整体置信度加权聚合（BLOCKING — 禁止主观判断）

报告的"整体置信度"必须按以下公式计算，不接受拍脑袋：

```
整体置信度得分 = 
    致命假设置信度平均 × 0.5
  + 核心假设置信度平均 × 0.3
  + 信源评分中位数 / 5 × 0.2

置信度转换：
  ✅高 → 1.0   ⚠️中 → 0.6   ❓低 → 0.3

阈值映射：
  得分 ≥ 0.80 → ✅ 高
  0.55 ≤ 得分 < 0.80 → ⚠️ 中
  得分 < 0.55 → ❓ 低
```

**示例计算**（深度尽调，3 致命 + 2 核心假设）：

```
致命：[✅高, ⚠️中, ✅高] → 平均 (1.0+0.6+1.0)/3 = 0.87
核心：[⚠️中, ⚠️中] → 平均 (0.6+0.6)/2 = 0.60
信源评分中位数：4（L2 权重）→ 4/5 = 0.80

整体得分 = 0.87×0.5 + 0.60×0.3 + 0.80×0.2 = 0.775
→ 映射为 ⚠️ 中置信度
```

必须在 `05_交叉验证记录.md` 中展示计算过程。

### 11.3 Pre-Mortem + Devil's Advocacy（BLOCKING）

```
"假设六个月后证明我的结论是错的——
  1. 最可能错在哪里？
  2. 我忽略了什么信息？
  3. 哪个假设其实最脆弱？
  4. 有没有完全相反的解释也能说通？"
```

**Devil's Advocacy**：刻意为相反结论构建最强论据，如论据同样有力则降低原结论置信度或标注"争议性结论"。

偏见级联检查（确认偏误/锚定偏误/幸存者偏误/光环效应/可得性偏误）详见 `references/bias-checklist.md`。

**📁 保存动作**：置信度计算 + Pre-Mortem 结果 + 偏见检查写入 `05_交叉验证记录.md`

## Step 14: 专业报告生成

根据调研类型选择模板（详见 `templates/report-structure.md`）。视觉规范严格遵守 `references/style-guide.md`；配图/图表/标题规范遵守 `references/report-guide.md`。

### 14.1 视觉硬规范

- **配色**：主色 McKinsey Blue `#005EB8` + 辅色 `#4A90A4` + 中性灰（≤4 色，灰色为数据图主角）
- **Takeaway Title**：所有 H2 章节标题、图表标题必须含数字或方向词（↑/↓/领先/落后/达/仅等），**禁止话题词**
- **数字格式**：金额带千分位（`1,018 亿元`）；百分比保留 1 位小数（`23.5%`）；增速带方向（`↑ 15.3% YoY`）
- **数据图 > 概念图**：数据图（Mermaid/表格/截图）≥ 70%，AI 概念图 ≤ 30%（仅封面/章节过渡）
- **作者署名**：封面页眉 + 页脚 + "参考资料"前的"关于本报告"章节必须包含作者信息（见 §14.5）
- 详细规范见 `references/style-guide.md` §1-§9

### 14.2 执行摘要卡（SCQA + 3 KPI 仪表盘）

报告顶部必须输出以下结构，**任一字段缺失 → Step 15 评审不通过**：

```markdown
> ### 📊 执行摘要（1 分钟通读）
>
> **Situation（共识）**：[读者已知的行业背景，1 句]
> **Complication（变化）**：[最近发生的关键变化/冲突，1 句]
> **Question（隐含）**：[这次调研要回答的核心决策问题，1 句]
> **Answer（建议）**：**[Go/No-Go/有条件 + 一句话理由]**（置信度 ✅高/⚠️中/❓低）
>
> ---
>
> | **[KPI1]** | **[KPI2]** | **[KPI3]** |
> |:-:|:-:|:-:|
> | [大数字+单位] | [大数字+单位] | [大数字+单位] |
> | [一行注释] | [一行注释] | [一行注释] |
>
> ---
>
> **核心发现**（按重要性降序，3-5 条）：
> 1. [Takeaway 句，含数字/方向，≤ 30 字]
> 2. [...]
> 3. [...]
>
> **核心风险**：[1-2 条最需关注的风险，具体可识别]
```

**KPI 仪表盘选择原则**：

- 每个 KPI 必须有简短注释（1 行，≤ 15 字）

### 14.3 报告通用结构（10 章节）

1. **封面**：标题（Takeaway Title 格式）+ 封面概念图 + 署名栏（见 §14.5）
2. **执行摘要卡**（SCQA + 3 KPI 仪表盘）
3. **假设验证结果**：所有假设的最终判定矩阵（含验证状态、置信度、关键证据）
4. **结构化分析章节**：按类型选框架
5. **武器贡献矩阵**：每个假设 → 用了哪些武器 → 缺口是什么
6. **持续监测建议**（Step 12 信息哨兵）
7. **盲区声明**：未覆盖方面、信息局限、建议补充路径
8. **关于本报告**（新增）：作者、方法论、调研周期、置信度公式、数据截止日（见 §14.5）
9. **参考资料**：分 L1-L2（权威）+ L3-L5（次要）两组
10. **延伸操作**：Skill 联动出口 + AI 调研 10 原则遵循清单（见 `references/research-principles.md`）

### 15.2 Evaluator Agent（CLASSic 五维打分）

起独立 Agent 读报告全文，输出：

```json
{
  "accuracy": 0-5,        // 证据充分性：核心结论是否都有 ≥2 信源？
  "structure": 0-5,       // 结构完整性：SCQA + 9 章节齐全？
  "readability": 0-5,     // 可读性：Anti-Slop 词汇数 = 0？Takeaway Title 达标？
  "visual": 0-5,          // 视觉：配图数达标？数据图 ≥ 70%？配色合规？
  "completeness": 0-5,    // 完整性：盲区、风险、延伸操作、武器贡献矩阵都有？
  "total": 0-25,
  "feedback": {
    "accuracy": "[具体问题或 OK]",
    "structure": "[具体问题或 OK]",
    "readability": "[具体问题或 OK]",
    "visual": "[具体问题或 OK]",
    "completeness": "[具体问题或 OK]"
  },
  "pass": true | false    // total ≥ 21 (平均 4.2) 且任一维度不 < 3 → pass
}
```

### 15.3 回灌机制

- **pass = true**：另存为 `FINAL_商业情报报告.md`（去除 `_v1_draft` 后缀）
- **pass = false**：
  - 将 feedback 作为 correction_hint 回 Step 14
  - 主 Skill 读 feedback，针对性修改
  - 最多 3 轮迭代；超过 → 标注"⚠️ 未通过独立评审（迭代 3 次）"，但仍交付
  - 每轮迭代保存为 `FINAL_商业情报报告_v2_draft.md`、`v3_draft.md`
