# 诊所HIS竞品选型决策框架：从保达到开源，九家厂商全景对比

> **声明**: 本报告基于公开信息、产品试用体验和开源代码分析，数据采集截止 2026-05-03。部分结论受信息来源限制，置信度已在文中标注。
> **阅读建议**: 若您正面临"自研 vs 采购""开源 vs 商业""单体 vs 微服务"的决策，建议直接跳至「五、决策框架与行动建议」。

---

## 一、市场格局：一个「双轨并行」的碎片化市场

中国诊所HIS市场呈现典型的「双轨结构」：

**轨道A：垂直专科深耕者**
- 代表：陕西亚创（中西医深度）、保达云诊所（综合诊所+专科）
- 特征：在特定区域或专科建立深度壁垒，客户关系稳固，但技术栈偏传统
- 风险：云化转型慢，AI能力弱，面临新一代产品的降维打击

**轨道B：平台型SaaS玩家**
- 代表：ABC诊所管家、丁香云管家、平安万家
- 特征：互联网基因强，免费/低价策略获取客户，快速迭代
- 风险：盈利压力大，部分产品已停止运营或战略收缩（如平安万家）

**轨道C（新兴）：AI原生与医共体**
- 代表：惠每云（AI能力）、东软云基层（区域医共体）
- 特征：以新技术或政策红利切入，试图绕过传统HIS的功能竞争
- 风险：AI医疗的合规与落地周期不确定，医共体依赖政策持续性

> **关键洞察**: 市场没有绝对领先者。Top 5 厂商合计市占率不足 30%，剩余 70% 由数百家区域小厂商和私有化部署方案瓜分。这意味着**新进入者仍有结构性机会**。

## 三、深度对标：保达云诊所 —— 「稳健但老旧」的典型样本

[[保达云诊所深度调研报告]] 对保达进行了全栈逆向分析，结论可概括为四个字：**稳健但老旧**。

### 3.1 保达的技术画像

| 维度 | 现状 | 评估 |
|------|------|------|
| **技术栈** | .NET MVC + LayUI + SQL Server | 2015年前后的主流选型，维护成本低，但扩展性差 |
| **架构** | 单体应用 + 私有化部署为主 | 稳定性高，云化和多租户改造难度大 |
| **前端体验** | 传统桌面式Web界面 | 功能完整但交互陈旧，学习曲线陡峭 |
| **移动端** | 有配套App但功能阉割 | 医生移动端体验弱于竞品 |
| **AI能力** | 无 | 无智能问诊、无辅助诊断、无处方审核 |
| **多租户** | 无 | 每个客户独立部署，运维成本高 |

### 3.2 保达的商业模式启示

- **收费模式**: 一次性买断 + 年度服务费，客单价中高
- **客户黏性**: 极高。诊所一旦上线，数据迁移和员工培训成本构成强锁定
- **增长瓶颈**: 依赖直销团队，获客成本高，难以突破区域天花板

### 3.3 对鑫港湾的启示

保达的「稳健」是鑫港湾需要吸收的（医疗系统稳定性第一），保达的「老旧」是鑫港湾可以攻击的差异化缺口：

1. **现代UX**: 简化操作路径，降低新员工培训成本
2. **云原生多租户**: 降低边际运维成本，支撑规模化扩张
3. **AI辅助**: 在病历书写、处方审核、库存预测等场景引入LLM能力

## 四、开源方案：可借鉴但不可依赖

[[开源HIS系统代码深度分析报告]] 分析了三款开源HIS，核心结论：**架构可参考，生产不可用**。

### 4.1 三款开源系统对比

| 系统 | 技术栈 | 架构特点 | 可借鉴点 | 致命短板 |
|------|--------|----------|----------|----------|
| **OpenHIS** | .NET Core + DDD + 微服务 | 领域驱动设计，患者域/挂号域/收费域分离 | **领域模型分层**、事件驱动架构、多租户隔离方案 | 缺乏真实医疗场景验证，边界条件处理不完善 |
| **hospital-management-system** | Java + Spring Boot + 代码生成器 | RBAC权限、数据范围权限 | 代码生成器提升CRUD效率、权限模型 | 生成代码缺乏业务语义，长期维护成本高 |
| **C#开源HIS** | WinForm + .NET Framework | 桌面端响应快、操作路径短 | 私有化部署场景下的界面设计思路 | 云化/移动端扩展能力几乎为零 |

### 4.2 鑫港湾可直用的设计资产

1. **领域模型**: 借鉴 OpenHIS 的「实体→聚合根→仓储→应用服务」分层，重构鑫港湾的核心业务模型
2. **事件驱动**: 用 Domain Event 解耦挂号、收费、药房、库存模块，降低耦合度
3. **多租户**: Schema-per-tenant 方案可直接参考 OpenHIS 的实现

### 4.3 开源无法补齐的缺口

- **医保对接**: 无开源实现，需自行对接各地医保局接口
- **AI能力**: 无LLM集成、无智能诊断
- **运维体系**: 日志、监控、告警薄弱
- **合规文档**: 等保、数据安全、医疗器械软件资质需自行补齐

> **关键结论**: 开源HIS是「架构参考书」，不是「生产发动机」。鑫港湾应以开源为参考自研核心系统，而非直接基于开源二次开发。

### 5.2 诊所选型清单（给使用者）

| 检查项 | 必问问题 | 及格线 |
|--------|----------|--------|
| **核心流程** | 能否不走完试用就完成挂号→收费全流程？ | 必须能 |
| **医保对接** | 是否支持本地医保？对接费用谁承担？ | 必须支持，费用透明 |
| **数据安全** | 数据存在哪里？能否导出？导出格式？ | 云SaaS需明确所有权，本地部署需备份方案 |
| **培训成本** | 新员工多久能独立操作？ | ≤3天 |
| **隐性费用** | 除了年费，还有哪些必付费用？ | 无重大隐性费用 |
| **切换成本** | 如果未来换系统，数据能否完整迁移？ | 提供标准导出格式 |

### 5.3 鑫港湾差异化路径（给建设者）

基于以上分析，鑫港湾若要在九家厂商中建立差异化坐标，建议考虑以下组合策略：

**路径A：「现代保达」—— 稳健+云化+体验升级**
- 吸收保达的「功能完整性」和「稳定性」
- 用云原生架构替代单体部署，支撑规模化
- 用现代UX降低培训和操作成本
- 适合：追求「比保达更好用、比互联网产品更稳」的中间地带

**路径B：「AI+诊所」—— 智能辅助为核心卖点**
- 病历智能书写（语音/模板+LLM）
- 处方智能审核（配伍禁忌、剂量警示）
- 库存智能预测（基于处方数据的药品补货建议）
- 适合：客单价承受能力强的中高端诊所、连锁品牌

**路径C：「专科纵深」—— 不做全科做专精**
- 中医专科：舌象/脉象记录、方剂库、煎药管理
- 口腔专科：牙位图、种植/正畸疗程管理、影像集成
- 医美专科：疗程设计、术前术后对比、客户生命周期管理
- 适合：在垂直领域建立「懂行」的品牌认知

**路径D：「基层医共体」—— 政策红利+区域垄断**
- 与县域卫健局合作，打包基层诊所HIS
- 深度对接区域医保和公卫系统
- 以项目制+长期服务费的组合盈利
- 适合：有政府关系资源的团队
