# EC工业化规范手册 — 核心方法论摘要

## Audience

技术管理者（Tech Lead/架构师/项目协调者），正在管理多人+AI Builder协作的软件项目，面临执行质量不一致、Builder跳步、上下文丢失等问题。阅读后可获得一套完整的工业化防呆流程设计方法论。

## Core Thesis

**将软件工程从"手艺"升级为"制造"是可能的**——通过执行卡片（EC卡）+ 四角色循环 + 六大防呆支柱 + 失败模式库，即使"小白"Builder也能按卡执行不出错，首次执行即可达到85分以上。

## Context

- **Reader pain point**: AI Builder（如飞书Agent、Cline）缺乏人类工程师的隐性判断力，经常出现跳步、技术栈违规、幻觉性完成等问题
- **Why this matters**: 交付质量 = 设计质量 × 执行质量 × 审查质量。任何一个环节为0，整体归零
- **Why now**: AI-assisted编码正在普及，传统"靠人把关"的质量体系已无法覆盖AI Builder的执行边界

## Key Findings

1. **EC卡 = 工厂作业指导书，不是需求文档** — 需求文档是"菜谱描述"，设计文档是"烹饪步骤"，EC卡是精确到每一步工具/参数/检验标准的作业指导书。17个强制字段 + Step 0四步前置校验锁死了执行路径。（`src_20260503_feab72b2` Ch1-4）

2. **四角色循环体系中协调者是唯一信息枢纽** — 设计者（对外调研）、协调者（中枢管理）、开发者（执行）、测试者（验证）四角色互不直接通信，所有信息经由协调者流转。这杜绝了"设计者直接找开发者改代码"的越权行为。（`src_20260503_feab72b2` Ch4）

3. **14种失败模式覆盖了AI Builder的主要缺陷类型** — F001技术栈违规、F003任务理解偏差、F004幻觉性完成、F010认知权重偏差（跳步）等14种模式均有明确的触发信号和防御措施。F033"表面问题修复陷阱"是最新补充的架构层防御。（`src_20260503_feab72b2` Ch13）

4. **"只测不修"铁律解决了测试者角色混淆** — 预-UAT阶段发现Bug后，测试者不得自行修复代码。必须先记录、升级协调者、由开发者修复。违反此铁律是F013失败模式。（`src_20260503_feab72b2` Ch27）

5. **20条核心铁律覆盖设计-执行-审查-变更全生命周期** — CF-003"禁止事项强制列明"、CF-007"禁止git add -A"、CF-014"80分以下必须打回"等铁律构成了一套可背诵、可检查的行为边界。（`src_20260503_feab72b2` Ch0）

## Call to Action

如果你的团队正在使用AI Builder辅助编码，从最小的切入点开始：为下一张EC卡添加Step 0A读卡确认（6个理解确认问题），让Builder在写代码前先复述它要做什么。这是成本最低、收益最高的防呆起点。

## Draft

EC工业化规范手册是鑫港湾HIS系统在多人+AI Builder协作中沉淀出的一套完整方法论。它的核心洞察是：**AI Builder不是人类工程师的替代品，而是需要更刚性的前置约束才能正确执行的"超级初级工"**。

手册的六大防呆支柱提供了从设计到交付的全链路约束：
1. 强制字段防呆 — 空字段=卡片不合格
2. 参考代码骨架 — Builder可扩展但核心结构不可改
3. 禁止清单 — 禁止直接操作DbContext/禁止跳过单元测试
4. 契约输出标准化 — API/数据库/事件均需OpenAPI 3.0/Schema格式
5. 失败模式库 — 14种已知失败模式，每种有防御措施
6. 环境锁定 — 代码仓库/开发环境/推送目标均锁定不可变

手册的36章覆盖从EC卡理论到安全编码红线到Sprint总结的完整软件工程生命周期。对技术管理者而言，它提供了一套可裁剪、可迁移的工业化流程模板——即使不采用全部36章，仅引入Step 0四步前置校验和失败模式库，就能显著降低AI Builder的执行偏差。
