# B2B SaaS 竞品调研标准化手册

> **版本**: v0.1 Draft
> **适用范围**: 任何需要系统性竞品分析的 B2B SaaS 产品决策场景
> **核心理念**: "人工定义问题 + AI 放大执行 + 交叉验证保真"
> **预期产出**: 竞品对比矩阵、差异化定位建议、选型决策框架

---

## 一、流程总览：四阶段九模块

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Phase 1    │ → │  Phase 2    │ → │  Phase 3    │ → │  Phase 4    │
│  问题定义    │    │  信息采集    │    │  深度验证    │    │  决策输出    │
│  (人工主导)  │    │  (AI辅助)   │    │  (人机交叉)  │    │  (人工收敛)  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      │                  │                  │                  │
   1.课题立项         3.线上广搜          6.实操体验          8.矩阵评分
   2.维度拆解         4.定向深挖          7.用户访谈          9.决策输出
                     5.信息归档
```

**人机分工比例**: Phase 1/4 以人为主（7:3），Phase 2/3 人机协作（5:5 到 3:7）。

## 二、Phase 1：问题定义（人工主导，0.5-1天）

### 2.1 课题立项检查表

在启动任何信息采集之前，必须回答以下五个问题：

| 检查项 | 问题 | 示例（诊所HIS场景） |
|--------|------|---------------------|
| **目标** | 这次调研要解决什么决策问题？ | 鑫港湾HIS应如何差异化定位？ |
| **边界** | 竞品范围是什么？排除什么？ | 9家轻量级诊所HIS，排除医院级HIS |
| **维度** | 从哪些角度评估竞品？ | 产品形态、定价、技术栈、区域覆盖 |
| **深度** | 每个维度需要多深？ | 必须亲自走完挂号→收费全流程 |
| **时效** | 结论有效期多久？何时更新？ | 3个月，季度跟踪医保政策变化 |

> **失败模式**: 未明确边界就开始搜索，导致信息过载、结论发散。参考 [[一堂-调研行动营启动_原文润色]] 中"真实选题需筛选复杂度"的教训。

### 2.2 维度拆解模板

将调研问题拆解为 **5-10个互补且部分重叠** 的维度。维度设计原则来自 [[Kimi 深度调研集群方法论 (Deep-Research-Swarm)]]：

- **互补性**: 合起来覆盖完整问题空间
- **重叠性**: ≥30% 重叠用于交叉验证
- **独立性**: 每个维度可由独立 Agent / 研究员执行

**标准维度库**（按 B2B SaaS 通用性排序）：

1. **产品功能图景**: 核心模块、功能边界、用户体验路径
2. **定价与商业模式**: 版本分层、年费区间、隐性成本、ROI叙事
3. **技术架构**: 技术栈、部署方式、扩展性、集成能力
4. **市场定位**: 目标客户画像、区域覆盖、垂直行业深度
5. **竞争动态**: 市场份额、融资情况、战略动向、生态合作
6. **用户评价**: 真实用户反馈、NPS、痛点高频词、 workaround 模式
7. **合规与政策**: 行业监管、数据安全、医保对接、资质要求
8. **供应链与生态**: 上下游整合、ISV生态、硬件兼容性

> **诊所HIS专用维度**: 在上述基础上增加「医保对接能力」「专科深度（中医/口腔）」「AI辅助诊疗」三个维度。参考 [[轻量级诊所HIS调研全清单]]。

## 三、Phase 2：信息采集（AI辅助，2-3天）

### 3.1 线上广搜（Route A：广域搜索）

**执行者**: AI Agent（自动化）
**输入**: 竞品清单 + 维度列表
**输出**: 结构化信息采集表

**采集渠道优先级**:
1. **官网与产品页**: 功能清单、定价页、客户案例（信任度：中，存在宣传夸大）
2. **官方演示视频/试用账号**: 真实界面录屏（信任度：高）
3. **应用商店/软件市场**: 用户评价、下载量、更新频率（信任度：中高）
4. **行业报告与媒体**: 第三方分析、融资新闻（信任度：中）
5. **社交媒体与论坛**: 真实用户吐槽、 workaround 分享（信任度：高但碎片化）

**信息采集表模板**:

```markdown
### {竞品名称} — {维度}

- **Claim**: 厂商声明的功能/优势
- **Source**: 信息来源URL
- **Date**: 信息采集日期
- **Excerpt**: 原始摘录（保持客观，不加工）
- **Context**: 上下文（如：官网首页、定价页、用户评论第3条）
- **Confidence**: 信任度（高/中/低/需验证）
- **Collector**: 人工或Agent标识
```

### 3.2 定向深挖（Route B：聚焦搜索 + Route D：文件增强）

**触发条件**: 广搜发现关键矛盾或信息缺口时
**执行者**: 人工研究员 + AI Agent 协作

**深挖策略**:
1. **技术验证**: 若竞品声称"微服务架构"，通过登录页JS逆向、API响应头分析验证实际技术栈。参考 [[保达云诊所深度调研报告]] 中的"登录页逆向+JS解析"方法。
2. **定价交叉验证**: 对比官网定价、销售报价单、用户实际支付记录，识别隐性费用。
3. **功能真伪验证**: 官网列出的功能可能未上线或体验极差，必须通过试用账号亲自验证。

### 3.3 信息归档

所有采集信息按以下结构归档到 KDO wiki：

```
30_wiki/
  entities/           ← 竞品厂商实体卡片（如：保达云诊所、ABC诊所管家）
  concepts/           ← 方法论和抽象发现
  decisions/          ← 阶段性决策记录
```

每条核心结论必须经过 **三步编译法**: Condense → Critique → Synthesize。参考 [[KDO Protocol]]。

## 四、Phase 3：深度验证（人机交叉，2-3天）

### 4.1 实操体验：核心路径必须亲自走一遍

这是发现"宣传功能 ≠ 实际好用"差距的唯一方法。参考 [[轻量级诊所HIS调研全清单]] 的核心结论2。

**必走流程**（以诊所HIS为例）：

| 步骤 | 操作内容 | 观察重点 |
|------|----------|----------|
| 1. 挂号 | 新建患者、选择科室医生 | 患者信息录入效率、重复建档检测 |
| 2. 接诊 | 医生站打开病历、填写主诉/诊断 | 病历模板丰富度、填写速度、中西医支持 |
| 3. 开方 | 开具西药/中成药/中药处方 | 药品库完整性、配伍禁忌提醒、AI辅助 |
| 4. 收费 | 生成收费单、医保结算 | 医保对接流畅度、支付方式覆盖 |
| 5. 发药 | 药房收方、库存扣减 | 库存预警、批号管理、效期提醒 |
| 6. 报表 | 查看经营统计、患者分析 | 数据维度、导出灵活性、可视化质量 |

**体验记录模板**:
```markdown
### {竞品} — 实操体验记录

- **体验日期**: YYYY-MM-DD
- **账号类型**: 免费试用/演示账号/真实诊所账号
- **流畅度评分**: 1-5（操作无阻塞感）
- **惊喜点**: 超出预期的设计
- **卡点**: 明显不便或功能缺失
- **workaround 观察**: 用户如何绕过产品限制
```

### 4.2 用户访谈：线下调研是线上数据的纠偏器

**样本选择标准**（解决 [[轻量级诊所HIS调研全清单]] 中的 Open Question）：

| 维度 | 分层要求 | 目的 |
|------|----------|------|
| **诊所规模** | 小型（1-2医）、中型（3-5医）、连锁（5+） | 覆盖不同付费能力和复杂度需求 |
| **专科类型** | 全科、中医、口腔、专科 | 验证专科深度 |
| **区域** | 一线、二线、县域 | 验证区域服务能力和医保差异 |
| **使用年限** | 新用户（<6月）、老用户（>2年） | 获取 onboarding 和长期稳定性反馈 |
| **切换经历** | 有/无切换其他HIS经历 | 获取对比视角和真实切换成本 |

**访谈提纲核心问题**:
1. 您每天最常用的3个功能是什么？最不常用的3个是什么？
2. 有没有哪个场景让您不得不绕过系统、用Excel或手工处理？
3. 如果明天必须换系统，您最大的顾虑是什么？
4. 您向同行推荐这个系统的可能性（0-10）？扣分原因？

### 4.3 交叉验证引擎

采用 [[Kimi 深度调研集群方法论 (Deep-Research-Swarm)]] 的四级置信度体系：

| 置信度 | 标准 | 处理方式 |
|--------|------|----------|
| **高** | ≥2个独立来源确认，且包含线下访谈 | 直接采纳 |
| **中** | 1个权威来源确认，或≥2个中等来源 | 采纳但标注来源 |
| **低** | 来源薄弱或单一未验证 | 暂不采纳，列入待验证 |
| **冲突** | 不同来源存在统计分歧或解释分歧 | 显式记录冲突，从不压制 |

## 五、Phase 4：决策输出（人工收敛，1天）

### 5.1 竞品对比矩阵

**评分维度**（按项目重要性加权）:

| 维度 | 权重 | 竞品A | 竞品B | 竞品C | ... |
|------|------|-------|-------|-------|-----|
| 核心功能完整性 | 20% | 4/5 | 5/5 | 3/5 | ... |
| 用户体验流畅度 | 20% | 3/5 | 4/5 | 4/5 | ... |
| 定价竞争力 | 15% | 5/5 | 3/5 | 4/5 | ... |
| 技术架构先进性 | 15% | 2/5 | 4/5 | 5/5 | ... |
| 客户服务质量 | 15% | 4/5 | 3/5 | 3/5 | ... |
| 差异化特色 | 15% | 3/5 | 4/5 | 2/5 | ... |
| **加权总分** | 100% | **3.6** | **4.0** | **3.5** | ... |

> **评分原则**: 每项评分必须有具体证据支撑，附在矩阵脚注中。

### 5.2 差异化定位建议

基于矩阵分析，输出「差异化坐标图」：

1. **X轴**: 价格区间（低端 → 高端）
2. **Y轴**: 功能深度（通用 → 垂直专科）
3. **气泡大小**: 市场份额/用户量
4. **标注自身产品位置**: 明确选择的差异化象限

参考 [[轻量级诊所HIS调研全清单]] 中总结的5种差异化路线：
- AI能力路线（惠每云）
- 免费/低价路线（ABC诊所管家）
- 供应链整合路线（平安万家）
- 专科深度路线（中医/口腔专科HIS）
- 区域医共体路线（东软云基层）

### 5.3 决策输出模板

```markdown
# {项目名称} 竞品调研结论报告

## 1. 调研概览
- 调研目标: ...
- 竞品范围: N家（列出名称）
- 调研周期: X天
- 置信度评估: 高/中/低

## 2. 市场格局总结
- 市场分层描述
- 关键趋势判断

## 3. 核心发现（按维度）
- 每个维度的Top 3发现
- 附证据来源

## 4. 竞品对比矩阵
- 表格

## 5. 差异化定位建议
- 选择的象限和理由
- 需避开的红海区域

## 6. 风险与待验证项
- 置信度低的结论
- 未解决的冲突
- 建议的后续跟踪动作

## 7. 附录
- 访谈记录摘要
- 信息采集原始表
```

## 六、质量门禁（Quality Gates）

在交付最终报告前，必须通过以下检查：

| 门禁 | 检查项 | 通过标准 |
|------|--------|----------|
| **QG1** | 目标对齐 | 报告结论直接回答 Phase 1 定义的决策问题 |
| **QG2** | 证据充分 | 每个核心论点有≥2个独立来源或1个线下验证 |
| **QG3** | 冲突披露 | 所有信息冲突已在报告中显式标注 |
| **QG4** | 时效声明 | 明确标注数据截止日和结论有效期 |
| **QG5** | 可行动性 | 输出包含至少3条可直接执行的策略建议 |
