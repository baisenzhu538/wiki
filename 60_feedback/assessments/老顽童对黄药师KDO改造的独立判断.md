## 一、总体评价

**黄药师的改造方向是正确的，但执行有偏差。**

核心问题：**基础链路（ingest→enrich→produce）还没跑通，就开始做高级功能。**

---

## 二、改造的核心价值

### 2.1 Phase 0-4 路线图清晰

| 阶段 | 内容 | 状态 |
|------|------|------|
| Phase 0 | 堵漏（中文句分割、lint修复） | ✅ 方向正确 |
| Phase 1 | 核心体验（wiki合并、模板简化） | ✅ 方向正确 |
| Phase 2 | 知识网络（引用图谱、断链检测） | ✅ 方向正确 |
| Phase 3 | 文章生产线（模板、交付流程） | ⚠️ 模板过重 |
| Phase 4 | AI增强（LLM接口、智能分发） | ❌ 链路未通 |

### 2.2 Revision Tracking 是最大亮点

```
feedback → improve --apply → snapshot SHA-256 → 用户编辑 → revise --scan → 记录revision
```

这让KDO从"单向流水线"变成"能自我演化的知识系统"，这是质的飞跃。

**但问题在于**：这个系统依赖基础链路完好。基础链路不通，revision tracking也只是在空壳上打补丁。

### 2.3 LLM接口设计克制

- 零第三方依赖，用urllib.request
- 环境变量未配置时明确报错
- 不做静默降级

这是正确的工程哲学。

## 三、独立判断：存在的核心问题

### 问题1：源码和pip版本不同步（反复出现）

**现象**：`revision.py` 在pip安装版存在，但源码目录没有。

**历史**：这不是第一次。Round 2的enrich功能也出现过类似问题。

**根因**：没有建立"改源码→重新pip install→验证"的流程。每次改动后，黄药师没有验证pip安装版的实际行为。

**后果**：
- 如果黄药师在源码目录继续开发，revision.py不会参与后续迭代
- 将来`pip install`覆盖时，revision.py可能丢失
- 老板老朱无法在源码目录看到revision系统

**建议**：
```bash
# 每次改动后必须执行
pip install -e /path/to/source --force-reinstall
kdo --help | grep revise  # 验证存在
```

---

### 问题2：Phase 4 AI增强方向存疑

**黄药师做了LLM接口，但老板老朱实测发现**：
- ingest结构丢失
- enrich无LLM失效
- produce空壳
- Research A/B缺位

**结论**：LLM接口做了，但没跑通，等于没做。

**这不是接口的问题，是链路串联的问题。** 黄药师应该先：
1. 用最简单的方式（curl调用LLM API）跑通ingest→enrich→produce一次
2. 验证链路通了，再谈智能化和自动化

---

### 问题3：Phase 3文章模板过于复杂

黄药师的文章模板有8个区块：
```markdown
## 核心命题（Core Thesis）
TODO: 用一句话说明这篇文章要证明什么...

## 背景（Background）
TODO: 读者需要知道什么才能理解本文？

## 关键数据（Key Data）
TODO: 支持核心命题的关键数字、事实...

...（还有5个区块）
```

**问题**：对于"快讯"场景（街顺APP动态、行业快讯），这个模板太重了。

**建议**：区分模板复杂度
- **快讯模板（3区块）**：核心命题 + 一句话事实 + 行动建议
- **深度分析模板（8区块）**：完整论证结构

---

### 问题4：两套搜索并存

- `search_documents()` —— 简单关键词搜索
- `SearchIndex` —— 向量语义搜索

**问题**：`kdo query` 实际用的是 `search_documents()`，SearchIndex虽然做了但没被调用。

**结果**：两套都要维护，实际只用到一套。浪费开发资源。

## 四、优先级建议

| 优先级 | 内容 | 理由 |
|--------|------|------|
| **P0** | 解决源码/pip同步问题 | 现在改的代码随时可能丢失 |
| **P0** | 跑通ingest→enrich→produce单次闭环 | 老板实测发现链路不通 |
| **P1** | 中文句分割修复 | 现在中文资料完全处理不了 |
| **P1** | wiki合并模式 | 调研报告多，碎片化严重 |
| **P2** | Revision Tracking完整实现 | 设计很好，但先要跑通基础链路 |
| **P2** | 简化文章模板 | 快讯场景不需要8区块 |
| **P3** | SearchIndex集成到kdo query | 先用起来，再谈智能化 |

## 五、核心判断

> **黄药师的改造是"正确的方向，局部的完成"**

**正确**：
- 设计文档质量高，路线图清晰
- 源码实现基本到位，细节有疏漏
- Revision Tracking设计精妙，是KDO进化的关键拼图

**偏差**：
- 基础链路（ingest→enrich→produce）还没跑通，就开始做高级功能
- 多个功能并行开发，没有明确的"单链路串通"里程碑
- 文档写得比代码跑得通
