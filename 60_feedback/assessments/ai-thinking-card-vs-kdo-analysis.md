## 背景

2026-05-15，收到一份用「认知升级系统 v3.2」生成的 AI 思维卡 HTML 文件——对方读了 Russell & Norvig《人工智能：一种现代方法》后产出的深度阅读笔记。经复盘发现，对方的工作流是在 KDO 卡片结构**之上**叠加了一层"认知补丁安装"加工层。本评估分析两套流程的差异，并提出可落地的改进建议。

## 一个关键前提：他们是在 KDO 基础上加工的

对方的卡片结构（Summary → Condense → Critique → Synthesis）与 KDO 卡片完全一致。他们的 EVIDENCE / CONTRAST / ACTION / ICAP / FOLLOWUP 模块是在 KDO 骨架**之上**叠加的"深度加工层"，不是替代关系。

**这意味着：KDO 的原子化卡片 + 图链接架构是对的。对方证明了在这个骨架上可以长出更深的肉。我们不需要改骨架，需要加肉。**

## KDO 核心优势（不能丢的）

| 优势 | 说明 | 为什么不能改 |
|------|------|------------|
| **原子化卡片** | 一张卡 = 一个独立知识单元 | 可被任意组合引用，不会像单文档那样"改一处要同步 N 处" |
| **wikilink 图遍历** | `[[双链]]` + `related:` 边 + `prerequisites:` | agent 可沿图边横向跳转，发现交叉域关联——这是单文档做不到的 |
| **前端可查询** | `query_triggers` 字段支持自然语言检索 | 单文档只能全文搜索，无法做语义触发 |
| **增量更新** | 纠正一张卡不影响其他卡 | 单文档耦合——修正一个观点的成本是全篇重读 |
| **分域管理** | domain / map / module 三级分类 | 一张 AI思维卡 HTML 自包含但不可分类聚合 |

**一句话：KDO 是知识操作系统，AI思维卡是运行在这个系统上的一个应用。操作系统不能为了一个应用好看就改内核。**

## 对方加工层的 5 个新增模块 — 本质分析

对方在 KDO 骨架上新增的模块，本质是解决了 KDO 当前的一个盲区：**卡片把"是什么"讲清楚了，但没把"装上了没有"闭环掉。**

### 1. EVIDENCE（证据审计）

**他们做了什么**：不只列案例，强制做反事实检验 + 偏差识别 + 外部攻击搜集。

**KDO 当前对应**：`[Critique]` 节列局限性，但以作者自述为主，缺少系统性压力测试。

**本质**：这是知识质量的第二道门禁。KDO 有格式门禁（`kdo lint`），有来源门禁（`source_refs` 不可为空），但缺少**论证质量门禁**。

**建议**：在 `[Critique]` 节中增加一条结构化要求——**每张方法/框架卡必须包含至少 1 条外部攻击（该方法的反对者会说什么）**。不需要改成 EVIDENCE 的完整格式，只需要一个最小约束。

### 2. CONTRAST（边界地图）

**他们做了什么**：与既有方法对比冲突点/互补点 + "什么时候千万别用"的精确场景。

**KDO 当前对应**：`[Synthesis]` 有"关联卡片"但只列正向关联，不列排他条件。

**本质**：知识卡片的"适用条件"是和"核心内容"同等重要的一级信息。知道什么时候不用 = 真正理解了。

**建议**：`[Synthesis]` 增加一个固定的 "不要用的场景" 子节（2-3 条），不作为可选项而是必填。格式简单：`场景描述 → 为什么失效 → 替代方案`。

### 3. ACTION（场景触发 + 最小实验）

**他们做了什么**：三层落地——场景触发卡（"老板甩来新项目 → 别急，先问 P/E/A/S"）、新流程 SOP、最小可行实验。

**KDO 当前对应**：卡片停留在"知识描述"层。`[Synthesis]` 的学习建议偏宏观（"先学 A 再学 B"），不落地到具体动作。

**本质**：知识的最终验证是"行为有没有变"。对方把 ACTION 和信息本身放在同等地位，不是附录。

**建议**：方法/工具类卡片增加 `## Action Triggers` 节。格式：3 个触发场景 + 每个场景的第一个动作（一句话即可，不需要完整实验设计）。

### 4. ICAP（学习泵升路径）

**他们做了什么**：四级自我定位（Passive → Active → Constructive → Interactive）+ 下一步泵升计划。

**KDO 当前对应**：无此机制。卡片有 `difficulty` 字段但指向内容难度，不指向读者的掌握层级。

**本质**：这是一个"知识掌握度追踪器"。对个人知识库有价值，但对 KDO 核心架构不是必需的。

**建议**：**低优先级**。可以在 frontmatter 加一个可选的 `icap_level` 字段，但不需要作为卡片结构的强制模块。这个更像是一个独立功能（`kdo review`），不应该是卡片模板的一部分。

### 5. FOLLOWUP（复诊机制）

**他们做了什么**：三周后回头检查"哪些洞察完全没用过？哪些产生了意外价值？"

**KDO 当前对应**：无。卡片 `updated_at` 会记录修改时间，但没有"主动提醒复查"的机制。

**本质**：Learning Agent 的闭环验证——装补丁 → 等三周 → 检查装上了没有。

**建议**：这是一个系统层面的功能，不是卡片模板的改动。可以在 kdo 里加一个 `review` 子命令——对指定卡片做复诊提示。但这个优先级低于 1-3。

## 改进方案：分层推进

### 第一层：卡片模板最小改动（零系统成本）

适用于方法/工具/框架类卡片。概念类和索引类不受影响。

**改动 1**：`[Critique]` → 新增"外部攻击"约束

```
## [Critique]

### 内部局限性
（现有内容）

### 外部攻击
至少 1 条：该方法的反对者/竞争学派会怎么批评它？（不带 straw man）
```

**改动 2**：`[Synthesis]` → 新增"不要用的场景"

```
### 不要用的场景
| 场景 | 为什么失效 | 替代方案 |
|------|-----------|---------|
| ...  | ...       | ...     |
```

**改动 3**：新增 `## Action Triggers` 节

```
## Action Triggers

| 触发场景 | 第一个动作 |
|---------|-----------|
| 场景描述（用户实际会遇到的情况） | 马上可以做的 1 个动作 |
```

以上三个改动：
- 不破坏现有卡片结构（纯追加）
- 不影响 `kdo lint`（现有 lint 规则不检查这些节）
- 不影响非方法类卡片（概念/索引类不需要填这些）

### 第二层：新建模板卡（有模板但非强制）

新建 `yt-model-ai-thinking-card-template.md`，把对方的 8+2 模块做成一页 KDO 模板。当有人要做深度阅读加工时，复制模板即可。

好处：
- 保留了对方的完整 workflow（对有需求的场景）
- 不污染 KDO 标准卡片结构（模板是独立卡片）
- 模板本身是一张 KDO 卡片（可被引用、搜索、迭代）

### 第三层：系统功能（需要改 kdo 代码，低优先级）

1. `kdo review` 子命令 — 对指定时间前 enriched 的卡片做复诊提示
2. `icap_level` frontmatter 字段支持
3. 论证质量门禁（检测 Critique 是否含 boilerplate / 是否缺外部攻击）

## 不该学的

| 对方做法 | 不学的理由 |
|---------|-----------|
| **单文档 HTML 输出** | KDO 的价值在原子化 + 图链接。单文档好看但不可组合、不可遍历。如果要分享给外部读者，做成导出功能即可——不影响内部存储格式 |
| **精美 CSS 排版** | KDO 渲染层是 Obsidian。卡片内容是 markdown，排版交给 Obsidian 主题。不要为了好看把内容格式绑死 |
| **每本书都做完整 8+2** | 成本太高。只对"想改变操作系统"级别的输入做深度加工。KDO 的分层处理（快速入库 → 逐批富化）是对的 |
| **自包含叙事结构** | 对方的 MY TAKE / REFLECTION 是个人叙事，属于"日记"而非"知识卡片"。KDO 卡片追求可复用——我写卡是为了让一年后的自己（或 agent）能检索到。个人叙事放 daily note 或独立文件即可 |

## 最值得偷的三样东西（按 ROI 排序）

1. **"不要用的场景"成为必填**（ROI 最高）——成本是一张表，收益是区分"会用"和"真懂"。已在 Sprint 9 验证过 Constraints 去模板化的价值，这是同一方向的自然延伸。
2. **Action Triggers**（ROI 次高）——成本是 3 行触发场景 + 3 个第一动作。解决了 KDO 卡片当前从"知道"到"做到"的断层。
3. **证据审计的外部攻击约束**（ROI 第三）——成本是一条反对者视角。解决了 Critique 过于软性的问题。Sprint 9 的 Constraints 去模板化只做了"加一条特有边界"，没有做"从敌人视角找漏洞"——这是升级版。

## 开放问题（请欧阳锋判断）

1. 三个模板改动（Critique 加外部攻击、Synthesis 加不要用场景、新增 Action Triggers）——是先在 pilot 域（比如选 5 张 entrepreneur 卡）试改再推广，还是直接改模板然后新建卡自动带上？
2. 已有 130+ 张 yt-* 卡，这三个节加上去是只加新建卡还是回溯补？如果回溯，要不要做一个 sprint？
3. ICAP 层级标注——作为 frontmatter 字段还是作为 body 节？个人倾向于 frontmatter（可被 kdo 脚本读取统计），但这意味着又要跑一次 batch injection。
