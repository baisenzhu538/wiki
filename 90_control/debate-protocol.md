# 协作辩论协议

> 目标：Architect + Builder 的辩论不消耗用户的注意力。用户只做方向性决策。

---

## 三角色分工

| 角色 | 职责 | 对用户承诺 |
|------|------|-----------|
| **用户** | 定方向、定优先级、定角度 | 只做选择题（≤3 选项），不读长篇辩论 |
| **Architect** | 审查产出、发现问题、提炼选项 | 交到用户手上的必须是压缩后的选择题，不是评估全文 |
| **Builder** | 执行 KDO 流水线（capture → ingest → enrich → produce → validate → ship） | 不主动发起架构讨论，有问题报给 Architect |

---

## 辩论流程

```
Builder 执行 → 产出内容
  │
  ├── Architect 审查
  │     ├── 没问题 → 直接入库（不打扰用户）
  │     └── 有问题 → Architect 写评估
  │           │
  │           ├── Builder 看评估
  │           │     ├── 同意 → 直接执行修正
  │           │     └── 有异议 → 在同一文件 append 回应
  │           │           │
  │           │           └── Architect 二次回应
  │           │                 ├── 接受 → 达成共识，执行
  │           │                 └── 仍有分歧 → 压缩为 ≤3 选项
  │           │                       │
  │           │                       └── 用户拍板（30 秒内可读完）
  │           │
  │           └── 决策记录写入同一文件
  │
  └── 用户日常只收到：
        - 状态摘要："本周进度：3/200 门课已入库"
        - 选择题（当 Architect + Builder 无法达成共识时）
        - 方向性问题（"调研方法论文章往哪个角度写？"）
```

---

## 文件约定

- **一件事一个文件**。所有辩论轮次 append 在同一文件中，不另开新文件。
- 评估文件路径：`60_feedback/assessments/<主题>.md`
- 决策记录：在文件中用 `## 决策记录` 区块标记已关闭事项。
- 达成共识的事项直接执行，不经过用户。

---

## 评分校准规则

- 评分是诊断工具，不是考核。
- Builder 可以质疑评分，Architect 必须逐条回应（Accept / Amend / Reject）。
- 评分争议辩论后仍无法对齐的，由用户终决。

---

## 例外

以下情况可以直接找用户，不需先经过内部辩论：

1. Builder 被明确要求执行某任务但缺少方向输入（如"文章选什么角度"）
2. 优先级冲突（如"继续 enrich vs 先修 backlog"）
3. 外部变化（如课程源格式变更、API 不可用）
