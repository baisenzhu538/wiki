# 最佳实践池子：用一堆最佳实践打磨产品

> 审美工具箱第 2 张卡片。[[yt-model-pan-product-aesthetic-toolkit]] | [[yt-panproduct-aesthetic-collection]] | [[一堂]]

## Summary

最佳实践池子是审美工具箱的第二张卡片——核心操作是**黑盒思维**：把大量最佳实践放到一起，将产品扔进去，不断对比打磨、迭代，直到达到类似水准才算结束。从"搜集"到"池子"的关键跃迁：不是散落的素材，而是可以反复用来打磨产品的参照系。

## Claims

### 核心机制：黑盒思维

> "把大量最佳实践放到一起，将我的产品扔进去，不断迭代，直到达到类似水准才算结束。"

- 不分解要素，不做显性分析——纯对比、纯感觉
- 关键动作：集中搜集 → 筛选入围 → 对比打磨 → 建池子库

### 常见池子类型

| 池子类型 | 示例 |
|---------|------|
| 品牌名称 | 大量优秀的品牌名 |
| Logo/界面 | 大量优秀的 logo/界面设计 |
| 文章/音乐/演讲/课程 | 大量优秀的文章/音乐/演讲/课程 |
| ... | 任何可比较的产出物 |

### 建立池子的流程

```
搜集目标 → 搜集渠道 → 筛选入围 → 扔进池子 → 对比打磨 → 建池子库 → MECE整理 → 持续迭代
```

> 适合场景：所有渴望达到最佳实践的设计场景
> 进步方式：养成"用池子迭代产品"的习惯

## Visual Analysis

卡片的核心视觉是一个"池子"隐喻——产品被扔进大量最佳实践中间，通过对比被打磨。常见池子类型用并列列表呈现，暗示池子不是一个单一事物而是一个分类体系。底部的流程链用箭头连接，强调"建池子"不是一次性动作而是持续迭代的过程。

### 外部攻击：Dieter Rams的"少但更好" + Edward de Bono的"水平思考"

**Dieter Rams**（博朗首席设计师，"Ten Principles for Good Design"作者，被公认为20世纪最具影响力的工业设计师，苹果设计哲学的公认源头）的"少但更好"（Weniger, aber besser）设计哲学对"池子对比法"提出了减法维度的挑战。Rams论证：真正伟大的设计不是"比较了很多参考后组合出的最优"，而是"从众多可能性中删减到只剩下本质"的结果。Rams的挑战：把产品扔进大量最佳实践池子中不断对比打磨——这个操作假设"比较得越多设计越好"。但Rams会说：池子里每一个对照对象都在增加设计噪音——你在跟A比文案、跟B比交互、跟C比配色，每一项比出来的"差距"你都想去追——你的产品变得越来越像"池子中各取一点拼起来的弗兰肯斯坦"，而不是一个"从本质出发的、完整的、统一的设计"。

**Edward de Bono**（牛津大学心理学与医学博士，"Lateral Thinking"和"Six Thinking Hats"作者，创造力研究领域开创者之一）的水平思考理论对"对比打磨"提出了创造性方向的挑战。de Bono论证：大多数创造性解决方案不是通过对现有方案"逐步改进"达到的，而是通过"跳到系统之外"重新定义问题达到的。de Bono的挑战：池子对比打磨本质上是"垂直思考"——你在"现有的最佳实践"这个区域内通过反复比较来逼近最优。但de Bono会说：最突破性的设计来自"注意到池子本身没有覆盖的一个维度"——你不需要在池子内做到最好，你需要问：这个池子本身缺少了什么？什么是所有这些最佳实践都没有考虑的？——这个"池子外"的维度，才是真正创造差异的地方。

### 不要用的场景

| 场景 | 为什么失效 | 替代方案 |
|------|-----------|---------|
| 池子中的案例在你开始产品设计之前已经存在了一段时间——而这些案例在"被选为最佳实践"到"你开始对比"之间，消费者需求和市场条件可能已经发生根本变化 | de Bono的"池子外维度"警告：你的池子里的每一个案例都是对"当时那个条件下的最优解"——池子本身不会自动更新它缺失的维度。你在用2023年的"最佳产品名"的池子对比你2026年的产品命名→池子能告诉你的只是"相对于2023年的标准你是否够好"——但如果2026年的用户在用一个2023年完全不存在的标准来判断产品名（如"这个词在社交媒体上的搜索成本"），你的池子对比在跟你没有关系的维度上给你虚假的满足感 | 每季度做一次"池子维度审计"——不只看池子里的案例是否需要更新，看池子本身覆盖了什么维度、缺失了什么维度。邀请≥2个不同背景的人（非设计者/非圈内人）看你的产品，问他们"你在凭什么判断这个好不好"——如果他们提到了池子没有覆盖的判断维度，把那个维度补充进池子 |
| 产品设计的核心瓶颈不在"不知道什么是好的"而在"受技术/资源/时间限制做不出好的"——池子对比让你清楚看到"差距"但对"如何弥补差距"没有帮助 | Rams的"少但更好"和de Bono的水平思考都指向：对比式打磨的最大陷阱是让你聚焦在"别人做了我没做"的维度上——你被池子告知"那里比你好"，但你没被告知"为什么要做那个方向"和"在资源有限的情况下做什么方向投入产出比最高"。池子帮你诊断了差距但没有帮你排优先级 | 池子对比后强制执行"差距-可行性矩阵"——对每个池子揭示的差距评估两个分数：(1)这个差距对用户体验的影响有多大（1-5）；(2)我们缩小这个差距的可行性有多大（1-5）。用"影响×可行性"排序确定优先级——追可行性最高的≠追差距最大的 |

## Action Triggers

| 触发场景 | 第一个动作 | 成功指标 |
|---------|-----------|---------|
| 建了一个池子并在里面对比了自己的产品——感觉"每个案例都比我好"但不知道从哪个方向改 | 做"差值排序"——不是"谁最好"，是"我的产品跟池子中每一个案例的差距在哪一个具体维度上最大"。列出≥3个维度，在每个维度上给你的产品和池子中最好的3个案例打分——不要打"综合分"，打"这个维度上我离它还差多少" | 找到≥1个维度上"差距不大但改善后用户感知提升最大"的优先方向——不是追"差距最大的维度"（可能是一个你不具备改善条件的维度），是追"改善后用户最能感知到变化的维度" |
| 对比打磨了一轮后感觉产品"比以前好了"——但不确定是不是真的好了还是只是"更接近池子中的平均水平了" | de Bono会问：你变好的维度是池子本身覆盖的维度吗？如果是→你的产品正在"靠近池子"，但池子本身可能没有覆盖真正重要的维度。做"盲测"——把你的原版和改进版拿给≥3个没看过池子的人，只问"你觉得哪个更好？为什么？"——不让"接近池子"成为判断标准 | 盲测中至少发生一次"跟池子告诉你的相反"——有人觉得原版的某个特征比改进版好，而那个特征恰好是你在"靠近池子"中牺牲掉的 |
