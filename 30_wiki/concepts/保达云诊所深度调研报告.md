# 保达云诊所深度调研报告

## Summary

> **报告定位**: 鑫港湾HIS竞品深度调研（保达云诊所）
> **调研日期**: 2026-04-20
> **研究方法**: 登录页逆向 + JS解析 + 官网验证（无登录态Session）
> **核心判断**: 保达是"扎实但老了"的对手——20年积累让功能全，但架构重、无AI、无多租户SaaS能力。鑫港湾应以"AI辅助 + 多租户SaaS + 现代UX"打差异化，而非追求功能数量超越。

## Reusable Knowledge

### [Condense] 五条核心结论

1. **保达云诊所是.NET MVC单体架构的"legacy SaaS"典型**：ASP.NET MVC + LayUI（jQuery）前端，Cookie+Session会话，无JWT Token，无多租户Schema隔离。与鑫港湾（ASP.NET Core微服务 + Angular 17 + JWT）存在约5年代差，但功能完整性远超鑫港湾。

2. **九大核心模块覆盖诊所全链路**：患者管理、挂号管理、医生工作站、划价收费、药房管理、库房管理、统计报表、会员管理、微信生态。其中划价收费和库房管理是鑫港湾完全空白的短板（P0/P1优先级）。

3. **中医诊所专属功能是保达的核心差异化**：医馆挂号、理疗管理（针灸/推拿/拔罐/艾灸/刮痧）、代煎中心（中药代煎+物流配送）、中药划价（按帖/公斤/剂数）、膏方/丸剂处方模板、中药颗粒管理。这些是通用诊所管理软件不具备的垂直能力。

4. **医保对接是保达的隐性护城河**：内置广州/深圳/珠海/佛山/永州等多城市医保常量，鑫港湾尚未实现医保对接。在强政策驱动的医疗市场，医保能力直接影响诊所采购决策。

5. **鑫港湾的差异化窗口在于"现代架构 + AI"**：保达无AI能力、无多租户SaaS、UX停留在jQuery时代。鑫港湾若在完成基础闭环（划价/库房/报表）后，叠加AI辅助诊断和现代化用户体验，可在2-3年内形成代差优势。

### [Critique] 逐条质疑

**对结论1（保达是legacy SaaS典型）：**
- 前提假设：技术架构新旧直接等同于产品竞争力。.NET MVC虽然老旧，但保达20年积累的功能完整性和稳定性可能是诊所更看重的。基层诊所对"技术先进性"的感知远低于对"功能齐全+不出错"的感知。
- 边界与反例： LayUI + jQuery 在医疗行业仍有大量存量系统，维护成本和人才供给并不稀缺；保达的多城市医保对接能力恰恰证明了"老技术也能做好业务"。
- 可靠性评估：**高**——架构判断基于登录页HTML/JS逆向，技术栈识别准确。但"5年代差"的表述是工程师视角，不一定是客户视角。

**对结论2（九大模块覆盖全链路）：**
- 前提假设：保达的九大模块都达到了生产可用水平。但调研未拿到登录态Session，部分模块描述基于JS配置枚举和官网文档，未经验证真实交互流程。
- 边界与反例：模块存在≠模块好用。保达的界面停留在jQuery时代，操作效率可能远低于现代Angular/Vue应用；鑫港湾虽模块少，但UX现代化可能反而提升实际工作效率。
- 可靠性评估：**中**。模块列表可信度较高（有JS配置文件佐证），但功能深度和用户体验未经实测。

**对结论3（中医专属功能是核心差异化）：**
- 前提假设：中医诊所市场是保达的主要收入来源。但调研未提供保达的客户结构数据，无法验证"中医功能"在其整体业务中的权重。
- 边界与反例：如果鑫港湾的目标客户是综合诊所而非中医馆，则保达的中医功能对鑫港湾不构成直接竞争；反之，若鑫港湾计划切入中医市场，这些功能就是必选项。
- 可靠性评估：**中高**。功能存在性已验证（官网+JS配置），但战略重要性需结合客户结构数据确认。

**对结论4（医保对接是隐性护城河）：**
- 前提假设：多城市医保常量的内置意味着保达已与这些城市的医保系统完成对接认证。但"内置常量"不等于"实时结算通道打通"，可能只是前端界面预置了城市列表。
- 边界与反例：医保对接的政策门槛极高，且各城市接口差异大。保达若仅覆盖广东部分城市，则对全国市场的护城河有限；若覆盖城市数量持续扩张，则壁垒极高。
- 可靠性评估：**中低**。"内置医保常量"的证据明确，但"对接深度"未经验证。需要拿到保达的真实医保结算流水才能确认。

**对结论5（鑫港湾差异化窗口在现代架构+AI）：**
- 前提假设：诊所决策者愿意为"AI能力"和"现代UX"付费。但基层诊所的付费意愿和能力有限，AI功能的边际价值可能被高估。
- 边界与反例：保达的客户续费率若很高，说明其老技术+全功能的组合已满足市场需求；鑫港湾的AI差异化可能只在一线城市高端诊所有效，对三四线城市诊所无吸引力。
- 可靠性评估：**中**。技术趋势判断正确，但市场接受度存在不确定性。需要鑫港湾的客户访谈数据来验证。

### [Synthesis] 跨领域对标

**与现有概念的关联：**
- [[鑫港湾his系统分阶段整改报告]] — 保达调研是鑫港湾整改的竞品对标输入。鑫港湾的P0缺口（划价收费）和P1缺口（库房管理、统计报表）直接来源于保达的功能全景。
- [[互联网医院模式深度调研报告]] — 保达无互联网医院能力，而互联网医院牌照（A/B/C类）是诊所远程问诊的合规基础。鑫港湾若在补齐基础HIS后叠加互联网医院能力，可形成"基础HIS + 互联网医院"的双层壁垒，这是保达不具备的。
- [[诊所o2o外卖平台业务深度调研报告]] — 保达无O2O送药能力，但其微信生态入口（公众号预约）是O2O的前置条件。鑫港湾可将微信预约 → 在线问诊 → 电子处方 → 药品配送串联为完整闭环。
- [[HIS系统深度调研]] — 保达是"传统单体SaaS"的典型代表，而行业趋势是云原生微服务+数据中台。保达的legacy架构反而验证了"功能 completeness 可以掩盖技术债务"这一市场现实。

**与已有概念的矛盾/互补：**
- **矛盾**：互联网医院报告主张"诊所应轻资产入驻第三方平台"，而保达调研暗示"诊所需要功能完整的自有HIS系统"。二者在"诊所IT投入策略"上存在张力——是花钱买全套HIS，还是省钱用第三方平台？
- **互补**：与 [[紫鲸AI智能体工作流平台]] 形成互补——紫鲸AI是"AI原生SaaS如何定义新品类"，鑫港湾是"传统SaaS如何AI化升级"。保达的"无AI"状态恰恰为鑫港湾提供了"AI差异化"的空白市场。
- **互补**：保达的中医专属功能与 [[互联网医院模式深度调研报告]] 中的"中药配送合规路径"可结合——中医诊所的代煎中心 + 互联网医院的药品配送 = 完整的中医数字化服务链。

**可迁移场景：**
- **竞品调研方法论**：登录页逆向 + JS解析 + 官网验证的"无账号调研法"，可迁移至任何Web-based SaaS产品的快速竞品分析。尤其适用于资源有限、无法获取试用账号的初创团队。
- **legacy SaaS升级路径**："先补齐基础功能闭环，再叠加AI差异化"的两阶段策略，可迁移至任何面临成熟legacy竞品的市场进入者。
- **医疗SaaS医保能力评估框架**："内置医保常量≠实时结算通道"的区分标准，可用于评估任何医疗信息化产品的真实医保能力深度。

## Open Questions

1. 保达的真实客户数量和续费率是多少？调研报告未提供财务数据，无法验证"20年积累"的市场成果。
2. 保达的医保对接是"前端常量预置"还是"实时结算通道已打通"？这需要拿到保达的真实医保结算流水才能确认。
3. 鑫港湾的目标客户是否包含中医诊所？如果不包含，保达的中医专属功能对鑫港湾的战略意义有限。
4. 保达的定价策略和收费模式是什么？SaaS年费还是私有化买断？这直接影响鑫港湾的定价参考。
5. 保达的微信预约六步流程是否为行业标准？ABC诊所管家、尚医云等竞品是否采用相同流程？
6. 调研未拿到登录态Session，保达的后台管理功能和数据报表的真实交互体验如何？
