# 一堂方法论体系总图

> 一堂（yitang）是面向创业者和业务负责人的商业管理教育公司，六年间研发了 200+ 门课程，覆盖 120+ 门方法课和 80+ 门案例课。本文档是 一堂全部方法论课程的**总索引和导航 Hub**。
> 
> 📋 完整 Dataview 课程列表见 [[yitang-course-map]]（按 yitang.map 自动聚合）

## 一、四张地图总览

一堂将全部方法论课程组织为四张地图，对应中国儒家「修身、齐家、治国、平天下」的核心理念：

```
个人修炼（修身）→ 管理修炼（齐家）→ 创业修炼（治国）→ 无限修炼（平天下）
     ↓                    ↓                    ↓                    ↓
   天花板               段位                 概率                 笃定
```

| 地图 | 核心追求 | 核心命题 | 时间尺度 |
|------|---------|---------|---------|
| **个人修炼** | 天花板 | 跳出时间轴修身，借假修真——所有事都是假的，我变强才是真的 | 一生（从童年到退休） |
| **管理修炼** | 段位 | 能操盘多大规模的资源、实现多高难度的目标 | 1-5 年/项目周期 |
| **创业修炼** | 概率 | 在资源耗光前让 1% 的概率翻十倍 | 1-5 年/项目周期 |
| **无限修炼** | 笃定 | 忘掉工具和课程，回到客观世界的客观规律 | 无限 |

### 四张地图的辩证关系

1. **个人 vs 做事（左-右辩证）**：向右走是"在时间轴上做事"（管理+创业），向左走是"跳出时间轴修身"（个人）。随时可以切换——焦虑时往左看，行动时往右走。

2. **管理 vs 创业（两只手辩证）**：成长做新事，新事带新人。管理练到一定段位就进入创业（做新业务），创业验证出确定性就进入管理（带团队放大）。在十年尺度上，这两只手来回交替。

3. **具体 vs 抽象（量变-质变辩证）**：左边三张地图是"把书越读越厚"（量变，学得越多工具越多），右边 Y 模型是"把书越读越薄"（质变，所有课归结为科学方法）。初学靠左边积累，高阶靠右边顿悟。

## 二、个人修炼地图（修身）

**核心追求**：不断拉高个人成长的天花板，突破能力/认知的上限。

### 结构：顶天立地模型

```
        ┌─────────────────┐
        │   人生红点（上限） │  ← 认知高度决定上限
        ├─────────────────┤
        │   领先层（值钱）   │  表达力 · 设计力 · AI 能力
        ├────────┬────────┤
        │ 认知（建）│ 能力（练）│  ← 迭代层
        │ IPO/复盘 │ 刻意练习  │
        │ 知识管理 │ 清单体笔记 │
        │          │ 逐字稿等  │
        ├────────┴────────┤
        │  时间管理（下限）  │  ← 时间效能决定下限
        └─────────────────┘
```

### 学习姿势

> "不管创业还是职场，不管开心还是焦虑，不管成功还是受挫，每经历一件事都能让自己变强一大步。尽早锁定人生红点，并且坚定地相信成长是解决问题唯一的方法。"

## 三、管理修炼地图（齐家）

**核心追求**：提升管理段位——能操盘多大规模的资源、实现多高难度的目标。

### 结构：五级递进

```
管自己 → 管项目 → 管团队 → 管业务 → 管公司
  │        │        │        │        │
  │    定方案    招聘    管目标   战略文化
  │    拆计划    管理    业务公式   股权
  │    管执行    团建    ROI     财务
  │    做复盘   知识萃取  动力阻力  一号位
  │          一对一沟通  创新业务
```

### 学习姿势

> "管理能力的进步从来不是别人给你的 title，而是你管人理事的核心是修炼确定性。新人提前做储备，老人把它当作极好的诊断工具。"

### 三种使用方式
1. **储备式学习**：超前学 1-2 个段位，避免"职业震动期"
2. **从后往前复盘**：业务问题→团队问题→项目管理问题→个人问题
3. **团队诊断工具**：识别每个下属的长短板

| 2 | 转化率黑客 — 动力阻力触点 | 理解业务增长的核心逻辑 |
| 3 | 理解基本功 | 管理的基本认知框架 |

## 四、创业修炼地图（治国）

**核心追求**：控制概率——在资源耗光前让 1% 的创业成功概率翻十倍。

### 结构：四阶段

```
预判（选择题）→ 起盘（验证期）→ 增长（加速期）→ 底层逻辑
   │                │                │              │
 商业模式拆解    关键假设拆解    投放/选址     实事求是
 行业周期分析   五步法（左）    销售/增长飞轮   解放思想
 集中度分析    低成本验证（右）   组织能力      Y模型
 屏蔽风险      MVP测试         竞争优势
```

### 学习姿势

> "对于任何单一的决策来说，创业都是一个小概率游戏。我们训练的所有能力都是在资源耗光以前，让背着定时炸弹的自己从原始森林里跑出去。"

## 五、无限修炼地图（平天下）

**核心追求**：笃定——忘掉所有工具和课程，用客观规律直接解题。

### 结构：三阶段

```
实事求是 → 解放思想 → Y 模型/科学方法
```

### 课程清单

仅 5 个标签，统领左边三张地图。课程本身已列入创业地图底层逻辑。

| 标签 | 说明 |
|------|------|
| 实事求是 | 回到业务现场，拒绝闭门造车 |
| 解放思想 | 打破认知框架，拒绝路径依赖 |
| Y 模型/科学方法 | 三角度八参数，所有课程的统一底层 |

三节必修课建议 90-200 学分后修，是四张地图中最抽象但最统领的一张。

### 学习姿势

> "忘掉工具、忘掉课程、忘掉技术，回到最朴素的起点，学习客观世界的客观规律。我们掌握的规律越底层，越轻盈、越自信、越笃定。没有什么道路可以通向科学，因为科学本身就是我们唯一的道路。"

### 必修课（三节，建议 90-200 学分后修）

| 序号 | 课程 | 说明 |
|:--:|------|------|
| 1 | 务实创业 / 实事求是 | 回到业务现场找答案 |
| 2 | 解放思想 | 打破认知框架和路径依赖 |
| 3 | 科学理念与 Y 模型 | 用科学方法还原一切 |

## 六、十层解读框架

一堂创始人 Trumen 提出的理解课程体系的十个分析角度：

| 层 | 角度 | 核心问题 |
|:--:|------|---------|
| 1 | **全局** | 四张地图如何构成"修身→齐家→治国→平天下"的完整链条？ |
| 2 | **追求** | 每张地图的核心关键词是什么？（天花板/段位/概率/笃定） |
| 3 | **细节** | 每张地图内部的模块结构和依赖关系是什么？ |
| 4 | **姿势** | 每张地图的正确学习心态是什么？ |
| 5 | **辩证** | 四张地图之间的张力和统一是什么？ |
| 6 | **启动** | 新同学每个地图学哪三节必修课？ |
| 7 | **段位** | 老同学如何从消除盲区→管理学习→理解辩证→归因→团队辅导→突破创新？ |
| 8 | **打勾** | 在地图上标记学习进度，建立成长的视觉化正反馈 |
| 9 | **快捷通道** | 通过 course_id 快速进入选课中心 |
| 10 | **场景** | 战略会复盘、日常选课、团队诊断、上帝视角全局思考 |

## 七、学习路径建议

### 新同学（< 50 学分）

1. 每个地图学 3 节必修课（共 12 节）
2. 重点：IPO 科学学习 → 时间管理 → 刻意练习
3. 建立基本认知框架，消除"迷路感"

### 成长中（50-150 学分）

1. 逐模块深入学习感兴趣的地图
2. 开始建立课程之间的联系
3. 用笔记记录学习，写作业完成思考闭环

### 体系化（150-300 学分）

1. 四张地图全覆盖，消除盲区
2. 理解课程之间的辩证关系
3. 能向前置归因——用底层问题解释上层现象

### 高阶（300+ 学分）

1. 进入无限修炼阶段
2. 用 Y 模型还原整张地图
3. 能辅导团队，拿地图做诊断工具

## 欧阳锋管线探伤报告：ChatGLM 链接测试暴露的结构性问题（2026-05-06）

> 用户在 inbox 放入一个 ChatGLM 分享链接 `https://chatglm.cn/share/3FOowrEZ`，目的是探伤 KDO 管线本身。以下是测试发现和改进方案。

### 测试结果

| 管线节点 | 预期 | 实际 | 结论 |
|----------|------|------|------|
| `kdo capture <url>` | 自动抓取 URL 标题+正文 → `00_inbox/links/` | 手动创建了 capture md（仅含 URL + 备注） | capture 本身可走，但依赖手动 |
| `kdo fetch-url` | curl 拉取 HTML → 提取正文 | ❌ SPA 页面，curl 拿到空壳 | **结构性失败** |
| `kdo ingest` | 编译 inbox → raw sources + wiki 骨架 | ❌ Python 不可用 | **环境问题** |
| `WebFetch` | 外部 HTTP GET → 提取正文 | ❌ chatglm.cn 被安全策略拦截 | **网络限制** |

### 根因分析

两个独立瓶颈：

**瓶颈 A：SPA 页面是 KDO 抓取层的结构性盲区**

`kdo fetch-url` 依赖 `curl` → `html2text`，只能处理服务端渲染页面。ChatGLM、飞书文档、Notion 分享页等大量现代内容托管在 SPA 上，JS 执行后才能渲染正文。这不是偶发问题——未来一堂课程链接、学员分享链接大概率也是 SPA。

**瓶颈 B：Python 运行时不可用**

`.kdo/kdo.py` 存在于仓库中，但当前执行环境（WSL + PowerShell）均无 Python。所有 kdo 命令静默失败。

### 改进方案

#### 改进 1：SPA URL 感知 + 分流路由

在 `kdo capture` 阶段增加 URL 类型检测：

```
kdo capture <url>
  ├─ 非 SPA（普通网页）      → kdo fetch-url → 自动提取正文 → 00_inbox/links/
  ├─ SPA（chatglm/飞书/notion）→ 生成手动捕获模板 → 00_inbox/links/xxx.md
  │                             模板包含：URL、标题占位、手动粘贴区、操作指引
  └─ 未知                    → 保守策略：先 curl 尝试，失败则降级为手动模板
```

**SPA 平台识别表**（待扩展到 `90_control/source-registry.yaml`）：

| 平台 | URL pattern | 策略 |
|------|------------|------|
| ChatGLM | `chatglm.cn/share/*` | 手动模板 |
| 飞书文档 | `*.feishu.cn/docx/*`, `*.feishu.cn/wiki/*` | 手动模板 或 feishu API |
| Notion | `notion.so/*` | Notion API 或手动模板 |
| 腾讯文档 | `docs.qq.com/*` | 手动模板 |
| 微信公众号 | `mp.weixin.qq.com/*` | curl 通常可用 |

#### 改进 2：`kdo fetch-url` 增加 Playwright 模式（Phase 2+，低优先级）

长期方案：当检测到 SPA 时，使用无头浏览器渲染后提取文本。成本较高（需要 Node.js + Playwright 依赖），但能覆盖 95% 的现代内容平台。

```
kdo fetch-url <url> --render        # 新 flag，走 Playwright
kdo fetch-url <url>                 # 默认 curl，SPA 时提示 --render
```

**不急着做**：手动模板覆盖了当前需求。Playwright 模式纳入 KDO Roadmap，等批量链接场景真正出现时再启动。

#### 改进 3：手动捕获模板标准化

当 SPA URL 被识别时，自动生成的模板应该长这样：

```markdown
---
title: "TODO: 页面标题"
source_url: "<原始URL>"
captured_at: "YYYY-MM-DD"
status: pending-content
capture_method: manual-paste         # 新字段
platform: chatglm                    # 新字段
tags:
  - "#external-resource"
---

# TODO: 页面标题

**来源**: <平台名> 智能体 / 文档
**链接**: <URL>
**状态**: ⚠️ 待手动粘贴内容

## 操作指引

1. 在浏览器中打开上述链接
2. 选中全部对话/文档正文
3. 粘贴到下方的「内容」区域
4. 将 status 改为 `ready-for-ingest`
5. 运行 `kdo ingest`

---

## 内容

<!-- 在此粘贴完整内容 -->
```

`kdo lint` / `kdo ingest` / `kdo fetch-url` 全部依赖 Python。需在项目 CLAUDE.md 或 `.kdo/` 中记录 Python 路径，确保后续 Agent session 启动时可用。
